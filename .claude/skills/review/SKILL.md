# レビュースキル

## チェック項目

### 1. 用語統一チェック
- `docs/glossary.md`に定義された用語が正しく使われているか
- 初出時の表記ルール（日本語（英語）形式）が守られているか
- 未定義の専門用語がないか

### 2. 章間整合性チェック
- 前の章の最後と今回の章の導入が自然につながっているか
- 他の章で説明される概念への参照が正しいか
- 矛盾する記述がないか

### 3. 図表完全性チェック
- chapter-specに定義された図表がすべて含まれているか
- 各節に最低1つの図表があるか
- 図表番号が正しいか（図X.Y形式）
- キャプションが付いているか
- 本文中で図表が参照されているか

### 4. 技術正確性チェック（web_search使用）

**このチェックはtech-verifierサブエージェントがweb_searchツールを使って実施する。**

検証対象:
- OCI Generative AI Serviceの仕様（利用可能モデル、API、リージョン、料金体系、Function Calling対応状況）
- OCI各サービスの仕様（Resource Manager、OKE、Autonomous Database、Streaming、Queue等）
- MCP / A2Aプロトコルの仕様と現在のステータス（バージョン、Linux Foundation移管状況等）
- エージェントフレームワーク（LangChain, CrewAI, AutoGen, LlamaIndex等）の最新情報
- Terraformプロバイダーの仕様
- コード例（ある場合）の構文・API呼び出しの正しさ

検証プロセス:
1. 原稿から技術的な主張を抽出する
2. 各主張をweb_searchで公式ドキュメント・信頼できるソースと照合する
3. 不一致・古い情報・確認不能な記述を報告する

### 5. 文体・表記チェック
- 常体（である調）で統一されているか
- 1文の長さ（60字推奨、80字上限）
- 段落の長さ（3-5文推奨）
- 禁止表現が使われていないか

### 6. ページ配分チェック
- chapter-specの目標ページ数との乖離
- 節ごとのバランス

### 7. 理解度チェック問題
- 3〜5問含まれているか
- 章の内容を適切にカバーしているか
- 解答と解説が含まれているか
