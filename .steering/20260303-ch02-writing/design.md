# 執筆設計書

## 執筆アプローチ

第2章は第1章で概観したエージェントの構成要素のうち「ツール」と「メモリ/コンテキスト」を深掘りする章である。前半（2.1〜2.4）でツール側（Function Calling → ツール定義 → MCP）を扱い、後半（2.5〜2.7）でコンテキスト側（コンテキストウィンドウ → 記憶の分類 → コンテキストエンジニアリング）を扱う。この二部構成により、エージェントの「手」と「記憶」を対照的に学べる。

目標ページ数は18pであり、7節構成のため各節平均約2.5pが目安。ただしMCP関連（2.3, 2.4）はやや重めに、ツール定義（2.2）は表中心のためやや軽めに配分する。

**導入**: 第1章でエージェントは「ツールを使う」と説明した。では、LLMはどのようにツールを選び、使うのか。そのメカニズムがFunction Callingである。

**本文の構成**:
1. Function Calling（Function Calling）の名称と概要
2. 図2.1でシーケンスを解説

4. ツール定義スキーマの概要（JSON Schemaベース）
5. 具体例: OCI SDKのlist_instancesをFunction Callingで呼ぶ場合の流れ

## 章の導入

第1章の結びを受けて、構成要素のうちツールとコンテキストに焦点を当てることを宣言する。「エージェントの「手」であるツールと「記憶」であるコンテキスト。この二つがエージェントの実行能力を決定する」で導入する。

## 章の結び

ツールとコンテキストの仕組みを理解したことを確認し、「しかし、1つのエージェントで全てを処理しようとするとどうなるか」という問いかけで第3章へつなぐ。シングルエージェントの能力を十分に理解した上で、その「限界」に目を向ける流れを作る。
