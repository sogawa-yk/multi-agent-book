# 執筆設計書

## 執筆アプローチ

第4章は第II部の冒頭であり、本書の理論的な核心となる章。六つの協調パターンを「分類の軸 → 個別パターン解説 → 組み合わせ → 選定ガイド」の流れで体系的に解説する。各パターンは統一的な構成（アーキテクチャ → 構造と特性 → 適用例 → トレードオフ）で記述し、読者が比較しやすくする。

## 節ごとの設計

### 4.1 パターン分類の軸

**導入**: 六つのパターンを個別に解説する前に、分類の枠組みを提示
**本文の構成**:
1. 制御方式（集中型/分散型/事前定義型）
2. データフロー（直列/並列/動的）
3. エージェント関係（対等/階層）
4. 三つの軸の関連性

### 4.2 直列パイプライン

**導入**: 最もシンプルなパターンとして、Unix/Linuxのパイプとの類比で導入
**図表設計**:
- 図4.1: 4段階パイプライン（ヒアリング→設計→Terraform生成→デプロイ）、各段階間にデータラベル
**本文の構成**:
1. アーキテクチャの解説
2. 構造と特性（データフローの単純性、コンテキストの分離、段階的な変換）
3. OCI上での適用例（インフラ構築パイプライン）
4. トレードオフ（柔軟性の欠如、エラー伝播、レイテンシ累積）

### 4.3 並列ファンアウト/ファンイン

**導入**: 第3章の「並行処理の不在」を直接解決するパターン
**図表設計**:

**本文の構成**:
1. アーキテクチャの解説
2. ファンアウト（サブタスクの独立性、データの並列化 vs 観点の並列化）
3. ファンイン（マージ/投票/選択の三つの戦略）
4. トレードオフ（最遅ワーカー依存、コスト比例）

### 4.4 オーケストレーター型

**導入**: 最も柔軟なパターンとして、動的な実行計画の概念で導入
**図表設計**:

**本文の構成**:

2. オーケストレーターの三つの役割（計画、分配、統合）
3. 動的な実行計画（Claude Codeの例）
4. トレードオフ（メタレベルのコンテキスト爆発、予測困難性、デバッグの複雑性）

### 4.5 スーパーバイザー型

**導入**: オーケストレーター型との違いを明確にしつつ、監督と委任の構造で導入
**図表設計**:

2. オーケストレーターとの違い（制御の粒度、エスカレーション機構、ワーカーの自律性）
3. 階層構造の深さ
4. トレードオフ（判断能力依存、コミュニケーションオーバーヘッド、エスカレーション設計）

### 4.6 コレオグラフィ型

**導入**: 唯一の純粋分散型パターンとして、マイクロサービスとの類比で導入

3. マイクロサービスとの類比
4. トレードオフ（予測困難性、デバッグ困難、合意形成の複雑さ）

### 4.7 評価者ループ

**導入**: 第3章の「信頼性の壁」を直接解決するパターンとして導入
**図表設計**:

**本文の構成**:

### 4.9 パターン選定ガイド

**図表設計**:

## 章の導入

第3章の五つの限界を簡潔に振り返り、「マルチエージェントの協調方法は一様ではない」として六つのパターンの体系的解説を宣言。第II部の冒頭であることを明示。

## 章の結び

六つのパターンの要約（番号付きリスト）の後、「エージェント同士が実際にどのように通信するか」という問いを提起し、第5章（MCP、A2A等の通信プロトコル）へ橋渡し。

## 理解度チェック問題の設計

1. 三つの分類軸と各パターンの位置づけ（種類: 概念確認）
2. オーケストレーター型とスーパーバイザー型の違い（種類: 比較分析）
3. 評価者ループによる信頼性の壁の解決（種類: 因果関係の説明）
4. ユースケースに基づくパターン選定（種類: 判断問題）
5. パターンの組み合わせの具体例（種類: 設計問題）
