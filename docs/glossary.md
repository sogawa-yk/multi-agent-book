## エージェント基礎

### エージェント (Agent)
**定義**: LLMを推論エンジンとし、ツール呼び出しとプランニング能力を持つ自律的なソフトウェアエンティティ。チャットボットとは異なり、目標に向けて自律的にアクションを実行する。
**初出**: 第1章

### シングルエージェント (Single Agent)
**定義**: 単一のLLMとツール群で構成されるエージェント。1つのエージェントが全てのタスクを処理する構成。
**初出**: 第1章

### マルチエージェント (Multi-Agent)
**定義**: 複数のエージェントが協調してタスクを遂行するシステム構成。各エージェントが専門的な役割を持ち、分業・並行・検証を行う。
**初出**: 第3章

### エージェントランタイム (Agent Runtime)
**定義**: エージェントの実行ループを制御する基盤。ループ制御、エラーハンドリング、停止条件の管理を担う。
**初出**: 第1章

### ReActパターン (ReAct Pattern)
**定義**: Reasoning（推論）とAction（アクション）を交互に繰り返すエージェントの基本動作パターン。LLMが状況を分析し、次のアクションを決定し、結果を観察して再び推論するループ。
**初出**: 第1章

## ツール・プロトコル

### Function Calling
**定義**: LLMが利用可能なツール（関数）の中から適切なものを選択し、引数を指定して呼び出すメカニズム。
**初出**: 第2章

### MCP (Model Context Protocol)
**定義**: LLMアプリケーションが外部ツールやデータソースに接続するための標準プロトコル。ツール提供の標準化を目的とする。
**初出**: 第2章

### A2A (Agent-to-Agent Protocol)
**定義**: エージェント間の通信を標準化するプロトコル。Agent Card、Task、Message、Artifactの概念を定義し、エージェント同士の発見・通信・協調を可能にする。
**初出**: 第5章

### Agent Card
**定義**: A2Aプロトコルにおけるエージェントの自己記述メタデータ。エージェントの能力、エンドポイント、認証方式等を公開する。
**初出**: 第5章

## コンテキスト・メモリ

### コンテキストウィンドウ (Context Window)
**定義**: LLMが一度に処理できるトークン数の上限。この制約がエージェント設計における重要な制約条件となる。
**初出**: 第2章

### コンテキストエンジニアリング (Context Engineering)
**定義**: LLMに渡すコンテキスト（プロンプト、会話履歴、ツール結果等）の内容と構成を戦略的に設計する技法。何を入れ、何を捨てるかの判断。
**初出**: 第2章

### 短期記憶 (Short-term Memory)
**定義**: エージェントの実行中に保持される一時的な情報。主に会話履歴やワーキングメモリとして機能する。
**初出**: 第2章

### 長期記憶 (Long-term Memory)
**定義**: エージェントの実行をまたいで永続化される情報。RAG、ベクトルDB、外部ストレージ等で実現する。
**初出**: 第2章

### ブラックボード (Blackboard)
**定義**: マルチエージェントにおける共有メモリパターンの一つ。全エージェントがアクセスできる共有データ構造に情報を読み書きする方式。
**初出**: 第6章

## 協調パターン

### 直列パイプライン (Sequential Pipeline)
**定義**: エージェントが直列に接続され、前のエージェントの出力が次のエージェントの入力となるパターン。段階的な変換や承認フローに適する。
**初出**: 第4章

### 並列ファンアウト/ファンイン (Parallel Fan-out/Fan-in)
**定義**: タスクを複数のエージェントに同時に分配（ファンアウト）し、結果を集約（ファンイン）するパターン。独立した調査や同時検証に適する。
**初出**: 第4章

### オーケストレーター (Orchestrator)
**定義**: 中央のエージェントが動的にタスクを分解・分配し、サブエージェントの結果を統合するパターン。タスクの複雑さに応じて柔軟に制御する。
**初出**: 第4章

### スーパーバイザー (Supervisor)
**定義**: 階層的な監督構造を持つパターン。オーケストレーターとは異なり、監督と委任の関係性を重視し、エスカレーションの仕組みを持つ。
**初出**: 第4章

### コレオグラフィ (Choreography)
**定義**: 中央制御なしにエージェント同士が自律的に協調するパターン。各エージェントが独立した判断で他のエージェントと通信する。ピアツーピア型とも呼ばれる。
**初出**: 第4章

### 評価者ループ (Evaluator Loop)
**定義**: 生成→評価→修正のサイクルを繰り返すパターン。あるエージェントが出力を生成し、別のエージェント（評価者/批評者）がそれを評価・修正指示する。
**初出**: 第4章

## 設計概念

### Human-in-the-Loop (HITL)
**定義**: エージェントの処理フローの中に人間の判断・承認ポイントを組み込む設計手法。リスクの高い操作や最終判断で人間が介入する。
**初出**: 第7章

### 冪等性 (Idempotency)
**定義**: 同じ操作を複数回実行しても結果が変わらない性質。インフラ操作を行うエージェントにおいて特に重要な要件。
**初出**: 第7章

### フォールバック (Fallback)
**定義**: エージェントが障害やタイムアウトにより機能不全に陥った場合に、代替のエージェントにタスクを再割り当てする仕組み。スーパーバイザー型パターンにおける耐障害性の基本機構。
**初出**: 第4章

### フォールトトレランス (Fault Tolerance)
**定義**: システムの一部が故障しても全体として機能し続ける能力。エージェントの失敗を前提とした設計。
**初出**: 第7章

### グレースフルデグラデーション (Graceful Degradation)
**定義**: システム全体が停止するのではなく、利用可能な機能だけで処理を継続する設計パターン。一部のエージェントが障害を起こした場合でも、残りのエージェントで可能な範囲の結果を返す。
**初出**: 第7章

## OCI関連

### OCI Generative AI Service
**定義**: Oracle Cloud Infrastructure上で提供される生成AIサービス。Cohere Command R+、Meta Llama等のモデルをAPI経由で利用できる。
**初出**: 第8章

### OKE (Oracle Container Engine for Kubernetes)
**定義**: OCI上のマネージドKubernetesサービス。エージェントのホスティングやコンテナ化されたワークロードの実行基盤として活用する。
**初出**: 第11章

### Resource Manager
**定義**: OCI上のTerraform実行マネージドサービス。IaC（Infrastructure as Code）によるインフラ構築・管理をOCI上で実行する。
**初出**: 第11章

### Autonomous Database
**定義**: OCIのセルフマネージドデータベースサービス。エージェントの知識ベースとして、Vector Search、JSON、SQLの機能を活用する。
**初出**: 第11章

### Resource Principal / Instance Principal
**定義**: OCI上のリソース（インスタンス等）に割り当てられる認証方式。エージェントがOCI APIを呼び出す際の認証に使用する。APIキーの管理が不要になる。
**初出**: 第7章

## 一般技術用語

### RAG (Retrieval-Augmented Generation)
**定義**: 外部知識ベースから関連情報を検索し、LLMの生成時にコンテキストとして提供する手法。長期記憶の実現方式の一つ。
**初出**: 第2章

### IaC (Infrastructure as Code)
**定義**: インフラの構成をコード（Terraform HCL等）で定義・管理する手法。エージェントによるインフラ自動構築の基盤技術。
**初出**: 第3章

### イベントソーシング (Event Sourcing)
**定義**: 状態の変更をイベントとして記録し、イベントの再生によって任意の時点の状態を復元できるパターン。エージェントの状態永続化戦略の一つ。
**初出**: 第6章
