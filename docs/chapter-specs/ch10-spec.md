# 第10章 マルチエージェントシステムの構築 ― 仕様書

## 章の概要

**章の目的**: 第II部の理論と第9章のシングルエージェント構築の知識を統合し、OCI上で動作するマルチエージェントシステムを設計・構築する方法を体系的に示す。本書の核心となる章であり、読者がマルチエージェントシステムを自力で構築できるようになることを目指す。
**目標ページ数**: 28p
**所属する部**: 第III部 OCI上での実践

## 前提知識

本章を読む前に、読者が理解している必要がある概念:

- **第4章**: 6つの協調パターン（特にオーケストレーター型の詳細）
- **第5章**: エージェント間通信（MCP、A2A、同期/非同期）
- **第6章**: 状態管理のパターン（共有メモリ、永続化戦略）
- **第7章**: 設計原則（単一責任、粒度設計、Human-in-the-Loop、フォールトトレランス）
- **第9章**: シングルエージェントの構築方法（ReActループ、ツール実装、MCP公開、認証）

## 節の構成

### 10.1 構築するシステムの全体像

**概要**: 本章で構築するマルチエージェントシステムのユースケースと全体像を提示する。OCI上のインフラ構築・管理を題材として、複数のエージェントが協調して作業を行うシステムの目標と範囲を定義する。第12章のケーススタディへの導入も兼ねる。

**図表**:
- 図10.1: マルチエージェントシステム全体アーキテクチャ図

**キーポイント**:
- 構築するシステムのユースケース: OCI上のインフラ関連タスクを複数エージェントで分担
- システム全体のアーキテクチャ: オーケストレーター + 複数の専門エージェント
- 各エージェントの役割概要（詳細は10.2で設計）
- 本章の構築が第12章のケーススタディにつながることを示す

### 10.2 エージェントの分割設計

**概要**: 第7章で学んだ設計原則（単一責任、粒度設計）を適用し、マルチエージェントシステムのエージェント構成を設計する。どの専門知識をどのエージェントに担わせるか、エージェント間のインターフェースをどう定義するかを決定する。

**図表**:
- 図10.2: エージェント分割設計図（各エージェントの責任範囲とインターフェース）
- 表10.1: エージェント一覧（名前 / 役割 / 入力 / 出力 / 使用ツール）

**キーポイント**:
- 分割の判断過程: なぜこの粒度で分割するのか、他の分割案との比較
- 各エージェントの責任範囲の明確化
- エージェント間のインターフェース定義: 入力・出力の形式、期待される振る舞い
- 第7章の単一責任の原則と粒度設計の実践的適用

### 10.3 オーケストレーターの実装

**概要**: マルチエージェントシステムの中核であるオーケストレーターの設計と実装を扱う。タスクの分解、サブエージェントへの委譲、結果の統合というオーケストレーターの責務を実装する。

**図表**:
- 図10.3: オーケストレーターのタスク分解・委譲フロー

**キーポイント**:
- オーケストレーターの三つの責務: タスク分解、委譲、結果統合
- タスク分解の実装: LLMによる動的なタスク分解 vs 事前定義されたワークフロー
- サブエージェントの選択ロジック: Agent Cardの活用
- 結果統合時の矛盾検出と解消
- 第4章のオーケストレーター型パターンの実装

### 10.4 エージェント間通信の実装

**概要**: エージェント間のメッセージ交換を実装する。同期通信と非同期通信の使い分け、メッセージフォーマットの設計、通信エラーへの対処を整理する。

**図表**:
- 図10.4: エージェント間シーケンス図（オーケストレーター↔サブエージェント間の典型的なメッセージフロー）
- 表10.2: 通信パターンの選択基準

**キーポイント**:
- MCP経由のツール呼び出しとしてのエージェント間通信
- A2Aプロトコルの適用: Agent Card公開、Taskベースの非同期通信
- 同期通信（即時応答）と非同期通信（長時間タスク）の使い分け
- メッセージフォーマットの設計: 構造化された入出力の定義
- 第5章の通信理論の実装

### 10.5 共有状態の実装

**概要**: 複数エージェント間で共有する状態の実装を扱う。タスクの進捗管理、中間成果物の共有、コンテキストの受け渡しを、OCI上のサービスを使って実現する。

**図表**:
- 図10.5: 共有状態の実装アーキテクチャ図（OCIサービスを使った状態管理）

**キーポイント**:
- 共有状態の実装先: Autonomous Database（構造化状態）、Object Storage（成果物）、Cache with Redis（一時状態）
- タスクの進捗管理: 各エージェントのステータスを一元管理
- 中間成果物の共有: あるエージェントの出力を別のエージェントの入力として受け渡す
- コンテキスト分離: 第6章で学んだ漏洩防止の実装
- 第6章の状態管理理論の実装

### 10.6 Human-in-the-Loopの実装

**概要**: マルチエージェントシステムに人間の承認ポイントを組み込む実装を扱う。リスクの高い操作の前に人間の判断を介在させ、承認後に処理を再開するフローを実現する。

**図表**:
- 図10.6: Human-in-the-Loopの実装フロー（承認リクエスト→待機→承認/拒否→再開）

**キーポイント**:
- 承認が必要な操作の判定ロジック
- 承認リクエストの通知: OCI Notifications / メール / Webhook
- 承認待ちの状態管理: タスクの一時停止と状態の永続化
- 承認後の再開: チェックポイントからの処理再開
- 第7章のHuman-in-the-Loop設計原則の実装

### 10.7 OCI上のホスティング

**概要**: マルチエージェントシステムをOCI上でホスティングするための構成を整理する。各エージェントの配置先（Container Instances、OKE、Functions等）、スケーリング戦略、可用性の確保を設計する。

**図表**:
- 図10.7: OCIインフラ構成図（各エージェントの配置とネットワーク接続）
- 表10.3: ホスティングオプション比較（Container Instances / OKE / Functions）

**キーポイント**:
- エージェントのコンテナ化: 各エージェントをコンテナイメージとしてパッケージング
- ホスティング先の選択: Container Instances（シンプル）、OKE（スケーラブル）、Functions（イベント駆動）
- オーケストレーターとサブエージェントの配置パターン
- スケーリング戦略: 負荷に応じたエージェントのスケールアウト

### 10.8 ネットワーク設計

**概要**: マルチエージェントシステムのネットワークレベルの設計を扱う。VCN内のエージェント間通信、インターネットへのアクセス制御、セキュリティリストの設計を整理する。

**図表**:
- 図10.8: ネットワーク設計図（VCN、サブネット、セキュリティリストの構成）

**キーポイント**:
- VCNの設計: パブリックサブネットとプライベートサブネットの使い分け
- エージェント間通信のネットワーク経路: 同一VCN内の通信、Service Gateway経由のOCIサービスアクセス
- セキュリティリスト / Network Security Group: エージェント間の通信を必要最小限に制限
- NAT Gateway / Service Gateway: エージェントからの外部アクセスの制御
- 第7章のセキュリティ設計原則のネットワークレベルでの適用

## 章末の理解度チェック問題

1. マルチエージェントシステムのエージェント分割設計において、分割の粒度を決める際の判断基準を三つ挙げ、それぞれを説明せよ。
2. オーケストレーターがタスクを分解する方式として「LLMによる動的分解」と「事前定義ワークフロー」があるが、それぞれの利点・欠点を比較せよ。
3. エージェント間通信で同期通信と非同期通信を使い分ける基準を、具体的なシナリオを挙げて説明せよ。
4. マルチエージェントシステムにおいてHuman-in-the-Loopを実装する際、承認待ちの状態管理をどのように設計するか。障害耐性を考慮して述べよ。
5. OCI上でマルチエージェントシステムをホスティングする場合、Container InstancesとOKEのどちらが適しているか。システムの規模・要件に応じて判断基準を述べよ。

### 接続のポイント

- **第II部（第4〜7章）から**: 第II部で学んだ理論（協調パターン、通信、状態管理、設計原則）が、本章で具体的なOCI上の実装として結実する。各節が第II部の特定の章に対応しており、「理論→実装」の対応が明確になるよう構成する
- **第9章から**: 第9章で構築したシングルエージェント（MCPサーバーとして公開済み）を、本章で複数連携させる。「シングル→マルチ」の拡張を段階的に示す

- **第13章へ**: 本章で構築したシステムを対象に、第13章でテスト・デバッグの方法を学ぶ。テスト可能な設計（ツールのモック化、通信のスタブ化等）は本章で意識しておく
