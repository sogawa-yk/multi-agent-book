# 第11章 OCI固有のサービスを活用するエージェント設計 ― 仕様書

## 章の概要

**章の目的**: OCI Generative AI Service以外のOCIサービス群（Resource Manager、OKE、Autonomous Database、Streaming/Queue、Logging/Monitoring、Vault、API Gateway）をエージェントのツールとして活用する設計パターンを体系的に整理し、エージェントが「OCIネイティブ」に動作するための知識を提供する。
**目標ページ数**: 20p
**所属する部**: 第III部 OCI上での実践

## 前提知識

- OCIの基本的なサービス（コンピュート、ネットワーク、IAM等）の理解
- 第8章で扱うOCI Generative AI Serviceの利用方法
- 第2章で扱うFunction CallingとMCPの仕組み
- 第4章で扱う協調パターンの基本概念
- 第9章・第10章で扱うOCI上でのエージェント構築の基礎

## 節の構成

### 11.1 Resource Manager ― IaCエージェントの基盤

**概要**: OCI Resource Manager（Terraform実行マネージドサービス）をエージェントのツールとして活用する方法を解説する。エージェントがTerraform HCLを生成し、Resource Managerのスタック作成・プラン実行・適用までを自動化するパターンを示す。ジョブのライフサイクル管理、状態ファイルの扱い、ドリフト検出との連携を整理する。

**図表**:
- 図11.1: Resource Managerとエージェントの連携アーキテクチャ

**キーポイント**:
- Resource Managerはエージェントにとって「インフラ操作の実行環境」として機能する
- スタック作成→プラン→適用の3段階をエージェントのツールとして設計する
- プラン結果のパース（何が作成・変更・削除されるか）がエージェントの判断材料となる
- Human-in-the-Loopの組み込みポイント（適用前の承認）

### 11.2 OKE ― コンテナ化されたエージェントの実行基盤

**概要**: Oracle Container Engine for Kubernetes（OKE）をエージェントのホスティング基盤として活用する設計を解説する。エージェントのコンテナ化、Pod設計、スケーリング戦略、ヘルスチェックの設計を整理する。マルチエージェントシステムにおけるエージェント間通信にKubernetesのService/Ingressを活用するパターンも示す。

**図表**:
- 図11.2: OKE上のマルチエージェントデプロイメント構成図

**キーポイント**:
- エージェントをコンテナとしてパッケージングすることで、デプロイとスケーリングが容易になる
- 各エージェントを独立したPodとして配置し、Kubernetes Serviceで通信する
- エージェントの長時間実行タスクに対するタイムアウトとヘルスチェックの設計
- Horizontal Pod Autoscaler（HPA）によるエージェントのスケーリング戦略

### 11.3 Autonomous Database ― エージェントの知識基盤

**概要**: Autonomous Database（ADB）をエージェントの知識ベースとして活用する方法を解説する。Vector Search機能を使ったRAG基盤の構築、JSONドキュメントストアとしての活用、SQLによる構造化データへのアクセスの3つの活用パターンを示す。エージェントの長期記憶の永続化先としてのADBの位置づけを明確にする。

**図表**:
- 図11.3: Autonomous Databaseの3つの活用パターン（Vector Search / JSON / SQL）

**キーポイント**:
- Vector Search機能によりADB単体でRAGの検索基盤を構成できる
- JSONドキュメントストアとしてエージェントの状態や会話履歴を柔軟に永続化できる
- SQL問い合わせをツールとして公開し、エージェントが構造化データに自然言語でアクセスするパターン
- 第6章で扱った状態管理・メモリアーキテクチャのOCI上での具体的な実装先となる

### 11.4 Streaming / Queue ― 非同期エージェント間通信

**概要**: OCI Streaming（Kafka互換）およびOCI Queue Serviceをエージェント間の非同期通信基盤として活用する方法を解説する。第5章で扱った同期・非同期通信の概念を、OCIサービスを用いて具体化する。イベント駆動型のマルチエージェントアーキテクチャにおけるメッセージングパターンを示す。

**図表**:
- 図11.4: StreamingとQueueを使ったイベント駆動型エージェント間通信

**キーポイント**:
- Streamingはイベントストリーム型の通信に適し、複数コンシューマーへのブロードキャストが可能
- Queueはタスク分配型の通信に適し、exactly-onceの処理保証を提供する
- StreamingとQueueの使い分け基準（ファンアウト vs タスク分配）
- 非同期通信により、エージェントの処理時間の差を吸収し、システム全体のスループットを向上させる

### 11.5 Logging / Monitoring ― エージェントの可観測性基盤

**概要**: OCI Logging ServiceとMonitoring Serviceをエージェントの可観測性基盤として活用する方法を解説する。エージェントの思考過程・ツール呼び出し・判断結果をログとして記録し、メトリクスとして可視化する設計パターンを示す。第14章で扱う可観測性の理論的基盤を、本節でOCIサービスに具体化する。

**図表**:
- 図11.5: OCI Logging/Monitoringを使ったエージェント可観測性の構成図

**キーポイント**:
- Custom Logsにエージェントの思考過程・ツール呼び出し結果を構造化ログとして記録する
- Monitoring Serviceのカスタムメトリクスでエージェントのパフォーマンスを可視化する
- ログ検索（Logging Analytics）によるエージェントの行動トレースの実現
- アラーム設定によるエージェントの異常検知（エラー率、レイテンシ閾値）

### 11.6 Vault ― シークレット管理とエージェントの認証

**概要**: OCI Vaultをエージェントのシークレット管理基盤として活用する方法を解説する。APIキー、トークン、外部サービスの認証情報をVaultで一元管理し、エージェントがResource Principal経由で安全にアクセスするパターンを示す。

**図表**:
- なし（11.7のAPI Gateway構成図と合わせてセキュリティ全体像を示す）

**キーポイント**:
- エージェントがアクセスする外部サービスの認証情報をVaultで一元管理する
- Resource Principal + Vaultの組み合わせにより、コード内にシークレットを埋め込まない設計を実現する
- シークレットのローテーションとエージェントの再取得戦略
- マルチエージェント環境でのアクセス制御（エージェントごとに異なるシークレットへのアクセス権限）

### 11.7 API Gateway ― エージェントの外部公開と保護

**概要**: OCI API Gatewayをエージェントの外部公開エンドポイントとして活用する方法を解説する。認証・認可、レート制限、リクエスト変換の機能により、エージェントを安全に外部公開する設計パターンを示す。A2AプロトコルのエンドポイントをAPI Gatewayで公開するシナリオも含む。

**図表**:
- 表11.1: OCIサービス×エージェント機能のマッピング表

**キーポイント**:
- API GatewayはA2Aプロトコルのエンドポイント公開基盤として機能する
- 認証・認可ポリシーによるエージェントアクセスの制御
- レート制限によるLLM API呼び出しコストの抑制
- API Gatewayのログ統合によるリクエストの可観測性確保

## 章末の理解度チェック問題

1. Resource Managerをエージェントのツールとして活用する場合、スタック作成→プラン→適用の3段階のうち、Human-in-the-Loopを組み込むべきポイントはどこか。その理由と合わせて説明せよ。
2. Autonomous DatabaseのVector Search機能とJSONドキュメントストア機能を、エージェントのどのような用途にそれぞれ使い分けるか述べよ。
3. OCI StreamingとOCI Queueの特性の違いを踏まえ、マルチエージェントシステムにおけるそれぞれの適用場面を説明せよ。
4. エージェントのシークレット管理にOCI Vaultを使う利点を、Resource Principalとの組み合わせの観点から説明せよ。
5. OCI API Gatewayをエージェントの外部公開に使う場合、セキュリティ面で考慮すべき設定を3つ挙げよ。
