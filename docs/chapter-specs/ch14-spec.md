# 第14章 可観測性とガバナンス ― 仕様書

## 章の概要

**章の目的**: 本番運用に耐える監視・ガバナンスの設計を学ぶ。エージェントの挙動を継続的に把握し、統制するための仕組みを理解する。

**目標ページ数**: 16p

**所属する部**: 第IV部 運用と発展

## 前提知識

この章を読むために必要な前提:
- 第13章の内容（テストとデバッグ、トレーシング）
- 第10章の内容（マルチエージェントシステムの構成）

## 節の構成

### 14.1 エージェントの可観測性 ― 何を計測すべきか

**概要**: マルチエージェントシステムにおいて計測すべきメトリクスを整理する。従来のソフトウェアの可観測性との違いを示す。

**図表**:
- 図14.1: エージェント可観測性の3本柱（ログ、メトリクス、トレース）

**キーポイント**:
- エージェント特有の計測対象：推論品質、ツール選択の正確性、トークン使用量
- 従来の可観測性に「エージェントの意思決定」が加わる

### 14.2 ログ設計 ― 構造化ログ、OCI Loggingとの連携

**概要**: エージェントのログを構造化して設計し、OCI Loggingと連携する方法を解説する。ログレベル、ログフォーマット、出力先の設計を示す。

**図表**:
- 図14.2: 構造化ログのフォーマット例とOCI Logging連携図

**キーポイント**:
- 構造化ログ（JSON形式）により、検索・分析が容易になる
- エージェントの思考過程をログに含める

### 14.3 メトリクス ― レイテンシ、トークン使用量、成功率

**概要**: エージェントシステムの健全性を計測するためのメトリクスを定義する。主要メトリクスの収集と可視化の方法を示す。

**図表**:
- 表14.1: エージェントシステムの主要メトリクス一覧

**キーポイント**:
- レイテンシ：ユーザー体験に直結
- トークン使用量：コストに直結
- 成功率：信頼性の指標

### 14.4 アラートと異常検知 ― エージェントの暴走をどう検知するか

**概要**: エージェントの異常動作（暴走、無限ループ、過剰なAPI呼び出し）を検知するためのアラート設計を解説する。

**図表**:
- 図14.3: 異常検知のアラート設計図

**キーポイント**:
- エージェント特有の異常パターン（ツールの繰り返し呼び出し、コスト急増）
- アラートの閾値設計と通知チャネル

### 14.5 コスト管理 ― LLM API呼び出しコストの最適化

**概要**: LLM API呼び出しのコストを管理・最適化する手法を解説する。コスト構造の分析、予算管理、最適化手法を示す。

**図表**:
- 図14.4: コスト構造の内訳図

**キーポイント**:
- トークン使用量がコストの主要因
- モデル選択、コンテキスト最適化、キャッシュ活用による最適化

### 14.6 ガバナンス ― 監査ログ、承認フロー、変更管理

**概要**: エージェントシステムのガバナンスを確保するための設計を解説する。監査ログ、承認フロー、変更管理の仕組みを示す。

**図表**:
- 図14.5: ガバナンスフレームワークの全体図

**キーポイント**:
- 「誰が」「何を」「いつ」決定したかの追跡可能性
- エージェントの行動に対する説明責任

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 4問
- 問題の種類: 概念の確認、判断問題、設計問題

### 問題案
1. エージェントの可観測性が従来のソフトウェアの可観測性と異なる点を3つ挙げよ（種類: 概念の確認）
2. エージェントの「暴走」を検知するためのメトリクスとアラート条件を設計せよ（種類: 設計問題）
3. LLM API呼び出しコストを削減するための手法を3つ挙げよ（種類: 概念の確認）
4. エージェントの行動に対する監査ログに含めるべき情報を列挙せよ（種類: 判断問題）
