# 第14章 可観測性とガバナンス ― 仕様書

## 章の概要

**章の目的**: マルチエージェントシステムを本番環境で安全かつ持続的に運用するために必要な可観測性（ログ・メトリクス・トレース）の設計と、コスト管理・ガバナンス（アクセス制御、監査、コンプライアンス）のフレームワークを提供する。
**目標ページ数**: 16p
**所属する部**: 第IV部 運用と発展

## 前提知識

- 第13章で扱うテストとデバッグの基礎（特にトレーシングの概念）
- 第11章で扱うOCI Logging/Monitoring Serviceの基本的な活用パターン
- 第10章・第12章で扱うマルチエージェントシステムの設計と構築経験
- 可観測性（Observability）の基本概念（ログ、メトリクス、トレースの3本柱）

## 節の構成

### 14.1 エージェントの可観測性 ― なぜ特別な設計が必要か

**概要**: マルチエージェントシステムの可観測性が、従来のマイクロサービスの可観測性と異なる点を整理する。エージェントの「判断」が不透明であること、実行パスが動的に変化すること、LLMのトークン消費がコストに直結することなど、エージェント固有の課題を明示し、可観測性設計の全体方針を示す。

**図表**:
- 図14.1: マルチエージェント可観測性の全体設計図（ログ・メトリクス・トレースの3層）

**キーポイント**:
- マイクロサービスの可観測性との共通点と差異
- エージェントの「思考過程」を可視化する必要性（ブラックボックス問題）
- 動的な実行パスの追跡（どのエージェントがどの順序で実行されたか）
- 可観測性の3本柱（ログ・メトリクス・トレース）のエージェントへの適用

### 14.2 ログ設計 ― エージェントの行動記録

**概要**: マルチエージェントシステムのログ設計を解説する。エージェントの思考過程、ツール呼び出し、判断結果を構造化ログとして記録する設計パターンを示す。ログレベルの設計、ログの構造化（JSON形式）、機密情報のマスキング、ログの保持期間とアーカイブ戦略を含む。

**図表**:
- なし（14.1の全体設計図で位置づけを示す）

**キーポイント**:
- 構造化ログの設計（エージェントID、実行ID、ステップ、ツール名、入力要約、出力要約）
- ログレベルの使い分け（DEBUG: LLM入出力全文、INFO: ツール呼び出し、WARN: リトライ、ERROR: 失敗）
- 機密情報のマスキング戦略（プロンプト内の個人情報、APIキー等）
- OCI Logging Serviceへの構造化ログの送信設計

### 14.3 メトリクス ― 何を計測すべきか

**概要**: マルチエージェントシステムで計測すべきメトリクスを体系的に整理する。エージェントレベル（レイテンシ、成功率、ツール呼び出し回数）、システムレベル（スループット、同時実行数、キュー深度）、ビジネスレベル（タスク完了率、ユーザー満足度）の3階層でメトリクスを設計する。

**図表**:
- 図14.2: 可観測性ダッシュボードの設計例

**キーポイント**:
- エージェントメトリクス（レイテンシ、成功率、LLMステップ数、ツール呼び出し回数）
- システムメトリクス（スループット、同時実行エージェント数、キュー深度、リソース使用率）
- ビジネスメトリクス（タスク完了率、Human-in-the-Loop介入率、平均処理時間）
- OCI Monitoring Serviceのカスタムメトリクスとしての実装

### 14.4 アラートと異常検知

**概要**: メトリクスに基づくアラート設計と、エージェント特有の異常検知パターンを解説する。静的閾値ベースのアラート、ベースラインからの逸脱検知、エージェントの「暴走」検知（無限ループ、過剰なAPI呼び出し）の設計を示す。アラート疲れを避けるための優先度設計と、エスカレーションフローを含む。

**図表**:
- なし（14.2のダッシュボード設計例にアラート設定を含める）

**キーポイント**:
- 静的閾値アラート（エラー率 > X%、レイテンシ > Y秒）
- エージェント暴走検知（ステップ数上限、トークン消費量上限、同一ツールの連続呼び出し検知）
- アラートの優先度設計（P1: サービス停止、P2: 品質劣化、P3: 警告）
- エスカレーションフロー（自動リカバリ → アラート通知 → 人間介入）

### 14.5 コスト管理 ― LLM API呼び出しのコスト最適化

**概要**: マルチエージェントシステムの運用コスト構造を分析し、コスト最適化の戦略を解説する。LLM API呼び出しのトークンコスト、エージェントの実行時間に伴うコンピュートコスト、ストレージコストの内訳を示し、コスト削減のための設計指針（キャッシュ、モデル選択、プロンプト最適化）を提供する。

**図表**:
- 図14.3: マルチエージェントシステムのコスト構造内訳図

**キーポイント**:
- コスト構造の可視化（LLM APIトークン、コンピュート、ストレージ、ネットワーク）
- LLM APIコストの最適化（プロンプト圧縮、不要なコンテキストの削減、キャッシュ戦略）
- モデル選択によるコスト最適化（タスクの複雑さに応じたモデルの使い分け）
- コスト予算とアラート（月次予算の設定、異常なコスト増加の早期検知）

### 14.6 ガバナンス ― アクセス制御・監査・コンプライアンス

**概要**: マルチエージェントシステムのガバナンスフレームワークを解説する。エージェントの権限管理（最小権限の原則）、操作の監査ログ、データプライバシー（LLMに送信するデータの管理）、コンプライアンス対応（操作記録の保持、承認フローの証跡）を整理する。OCI IAMとの統合によるガバナンスの実装パターンを含む。

**図表**:
- 表14.1: ガバナンスチェックリスト（カテゴリ × 対策 × OCI実装手段）

**キーポイント**:
- エージェントの権限管理（最小権限の原則、Dynamic Group + Policy による制御）
- 監査ログの設計（OCI Audit Serviceとの統合、操作記録の改ざん防止）
- データプライバシー（LLMに送信するデータの分類と管理、PII検出とマスキング）
- コンプライアンス（操作承認の証跡、データ保持ポリシー、リージョン制約の遵守）

## 章末の理解度チェック問題

1. マルチエージェントシステムの可観測性が、従来のマイクロサービスの可観測性と異なる点を2つ挙げよ。
2. エージェントの構造化ログに含めるべきフィールドを5つ挙げ、それぞれの目的を説明せよ。
3. マルチエージェントシステムのメトリクスを「エージェントレベル」「システムレベル」「ビジネスレベル」の3階層に分類し、各階層の代表的なメトリクスを1つずつ挙げよ。
4. マルチエージェントシステムのコスト構造において、最も大きな割合を占める傾向にあるコスト要因は何か。そのコストを最適化する手法を2つ述べよ。
5. エージェントに対して「最小権限の原則」を適用する際、OCI上でどのように実装するか説明せよ。

## 章間の接続

### 前の章との関係
- 第13章のトレーシング・デバッグ手法を、本番環境向けの可観測性設計へと発展させる
- 第13章が「テスト時の問題発見」に焦点を当てたのに対し、本章は「本番運用時の継続的な監視」に焦点を当てる
- 第11章で扱ったOCI Logging/Monitoring Serviceの活用パターンを、本章でより体系的なフレームワークに統合する
- 第12章のケーススタディで設計したシステムの運用に必要な可観測性・ガバナンスを本章で補完する

### 次の章（第15章: マルチエージェントの未来とエコシステム）への接続
- 本章で整理した運用上の課題（コスト、ガバナンス、可観測性）が、第15章の「現在の課題の整理」の基盤となる
- 本章のガバナンスの議論が、第15章の標準化動向の文脈へと拡がる
- 本章末で「マルチエージェントシステムの運用に必要な可観測性とガバナンスの設計を学んだ。最終章では、マルチエージェントの現在の課題を整理し、今後のエコシステムの発展を展望する」と接続する
