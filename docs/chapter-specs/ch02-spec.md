# 第2章 ツールとコンテキスト ― エージェントの「手」と「記憶」 ― 仕様書

## 章の概要

**章の目的**: Function Calling / MCP / コンテキスト管理を深く理解する。エージェントが外部世界と相互作用するための仕組みと、限られた記憶を効果的に使う戦略を学ぶ。

**目標ページ数**: 18p

**所属する部**: 第I部 シングルエージェントの深い理解

## 前提知識

この章を読むために必要な前提:
- 第1章の内容（エージェントの定義、構成要素、ReActパターン）
- REST APIの基本（HTTPメソッド、JSON）

## 節の構成

### 2.1 Function Callingの仕組み ― LLMがツールを選択するメカニズム

**概要**: LLMがどのようにツールを選択し、引数を生成するかのメカニズムを解説する。プロンプトにツール定義を含め、LLMが構造化された出力で応答する流れを示す。

**図表**:
- 図2.1: Function Callingのシーケンス図（ユーザー→LLM→ツール実行→結果返却の流れ）

**キーポイント**:
- LLMは「ツールを実行する」のではなく「ツール呼び出しの指示を生成する」
- ツール定義のスキーマがLLMの判断精度に直結する

### 2.2 ツール定義のベストプラクティス ― スキーマ設計、説明文の書き方

**概要**: LLMが正確にツールを選択・利用するためのツール定義の設計手法を解説する。名前、説明文、パラメータスキーマの書き方のコツを示す。

**図表**:
- 表2.1: ツール定義の良い例と悪い例の比較表

**キーポイント**:
- ツール名は動詞+名詞の組み合わせで明確に
- 説明文はLLMへの指示として書く

### 2.3 MCPの役割 ― ツール提供の標準化、なぜMCPが生まれたか

**概要**: MCP（Model Context Protocol）の背景と役割を解説する。ツール提供の標準化がなぜ必要とされ、MCPがどのような課題を解決するかを整理する。

**図表**:
- 図2.2: MCP以前と以後のツール連携の比較図

**キーポイント**:
- MCPは「LLMとツールの間のUSB-C」
- ツール提供の標準化により、再利用性と相互運用性が向上する

### 2.4 MCPのアーキテクチャ ― Transport層、プロトコル、ライフサイクル

**概要**: MCPの技術的なアーキテクチャを解説する。Transport層（stdio, HTTP/SSE）、プロトコルメッセージ、サーバーのライフサイクルを整理する。

**図表**:
- 図2.3: MCPアーキテクチャ図（クライアント-サーバー構成、Transport層）

**キーポイント**:
- MCPはクライアント-サーバーモデル
- Transport層の選択がデプロイ方式を決める

### 2.5 コンテキストウィンドウの管理 ― 限られた記憶でどう戦うか

**概要**: LLMのコンテキストウィンドウの制約と、それに対処するための戦略を解説する。トークン数の管理、情報の優先順位づけを扱う。

**図表**:
- 図2.4: コンテキストウィンドウの構成図（システムプロンプト、会話履歴、ツール結果の配分）

**キーポイント**:
- コンテキストウィンドウは有限リソースであり、設計対象である
- 何を入れるかだけでなく、何を捨てるかの判断が重要

### 2.6 短期記憶と長期記憶 ― 会話履歴、RAG、外部ストレージの使い分け

**概要**: エージェントの記憶を短期（コンテキストウィンドウ内）と長期（外部ストレージ）に分類し、それぞれの実装方法と使い分けを解説する。

**図表**:
- 図2.5: 短期記憶と長期記憶のアーキテクチャ図

**キーポイント**:
- 短期記憶はコンテキストウィンドウ内、長期記憶は外部に保存
- RAGは長期記憶から短期記憶への「検索転送」メカニズム

### 2.7 コンテキストエンジニアリング ― 何を入れ、何を捨てるかの設計

**概要**: LLMに渡すコンテキストを戦略的に設計する技法を解説する。情報の取捨選択、要約、構造化の手法を示す。

**図表**:
- 表2.2: コンテキスト管理の戦略マトリクス（情報の種類 × 保持/要約/破棄）

**キーポイント**:
- コンテキストエンジニアリングはエージェントの性能を決定的に左右する
- 「全部入れる」は最悪の戦略
