# 第2章 ツールとコンテキスト ― エージェントの「手」と「記憶」 ― 仕様書

## 章の概要

**章の目的**: エージェントが外部世界と接するメカニズム（Function Calling、MCP）と、情報を保持・管理する仕組み（コンテキストウィンドウ、短期・長期記憶、コンテキストエンジニアリング）を体系的に理解する。これらはシングル・マルチを問わずエージェント設計の根幹をなす。
**目標ページ数**: 18p
**所属する部**: 第I部 シングルエージェントの深い理解

## 節の構成

### 2.1 Function Callingの仕組み

**概要**: LLMがツール（関数）を選択・呼び出すメカニズムであるFunction Callingの仕組みを解説する。ユーザーのリクエストからLLMの判断、関数呼び出し、結果の返却までの一連のフローを明確にする。Function CallingがReActパターンにおける「行動」の具体的な実装であることを示す。

**図表**:
- 図2.1: Function Callingのシーケンス図

**キーポイント**:
- Function Callingは「行動」ステップの具体的な実装メカニズム
- LLMは関数の「定義」を見て適切な関数と引数を選択する
- 実際の関数実行はLLMの外部（ランタイム側）で行われる
- 並列Function Calling（一度に複数のツール呼び出し）の概念

### 2.2 ツール定義のベストプラクティス

**概要**: エージェントに与えるツールの定義方法と設計上のベストプラクティスを解説する。ツールの名前、説明文、パラメータスキーマの設計がLLMの判断精度に直結することを示す。過剰なツール提供の問題（ツール選択の困難さ）と、適切な粒度設計について述べる。

**図表**:
- なし（2.1のシーケンス図を参照しつつ、テキストで解説）

**キーポイント**:
- ツール名と説明文はLLMにとっての「取扱説明書」
- パラメータスキーマは厳密に定義する（型、制約、デフォルト値）
- ツール数が多すぎるとLLMの選択精度が低下する
- ツールの粒度は「一つのツールが一つの責務を持つ」が基本

### 2.3 MCPの役割

**概要**: MCP（Model Context Protocol）の目的と役割を解説する。MCPが解決する課題（ツール提供の標準化、ベンダー非依存のツール接続）を明確にし、MCPがエージェントエコシステムにおいてどのような位置づけにあるかを示す。

**図表**:
- なし（2.4のアーキテクチャ図で詳細を示す）

**キーポイント**:
- MCPは「ツール提供」を標準化するプロトコル
- MCPにより、ツール提供者とエージェント開発者が独立に開発できる
- MCPはFunction Callingの上位レイヤーとして機能する
- MCPは第5章で扱うA2Aとは異なる（MCPはツール提供、A2Aはエージェント間通信）

### 2.4 MCPのアーキテクチャ

**概要**: MCPの技術的なアーキテクチャを解説する。MCPホスト、MCPクライアント、MCPサーバーの3層構造、通信プロトコル（stdio、SSE/HTTP）、MCPが提供する3つのプリミティブ（Tools、Resources、Prompts）を整理する。

**図表**:
- 図2.2: MCPアーキテクチャ図（ホスト・クライアント・サーバーの3層構造）

**キーポイント**:
- MCPホスト（エージェントアプリケーション）がMCPクライアントを内包する
- MCPサーバーがツール・リソース・プロンプトを提供する
- 通信方式はstdio（ローカル）とSSE/HTTP（リモート）がある
- 3つのプリミティブ: Tools（関数実行）、Resources（データ提供）、Prompts（テンプレート）

### 2.5 コンテキストウィンドウの管理

**概要**: LLMのコンテキストウィンドウ（一度に処理できるトークン数の上限）がエージェント設計における最も重要な制約条件であることを解説する。コンテキストウィンドウの消費パターン（システムプロンプト、会話履歴、ツール定義、ツール実行結果）を分析し、管理戦略を示す。

**図表**:
- 図2.3: コンテキスト管理の戦略マトリクス（情報の重要度 vs 鮮度による分類）

**キーポイント**:
- コンテキストウィンドウはエージェント設計における最大の制約条件
- ツール定義、会話履歴、実行結果がコンテキストを圧迫する
- 長いタスクではコンテキストが溢れ、性能が劣化する
- この制約が第3章でのシングルエージェントの限界分析の伏線となる

### 2.6 短期記憶と長期記憶

**概要**: エージェントの記憶を「短期記憶」と「長期記憶」に分類し、それぞれの役割・実装方式・トレードオフを解説する。短期記憶はコンテキストウィンドウ内の会話履歴、長期記憶はRAGやベクトルDB等の外部ストレージによる永続化である。

**図表**:
- なし（2.5の戦略マトリクスと連携して解説）

**キーポイント**:
- 短期記憶: コンテキストウィンドウ内の情報（揮発性、高速アクセス）
- 長期記憶: 外部ストレージに永続化された情報（永続性、検索コスト）
- RAGは長期記憶の代表的な実装パターン
- 記憶の階層設計が第6章（状態管理とメモリアーキテクチャ）で深掘りされる

### 2.7 コンテキストエンジニアリング

**概要**: LLMに渡すコンテキストの内容と構成を戦略的に設計する技法であるコンテキストエンジニアリングを解説する。「何を入れ、何を捨てるか」の判断フレームワークを提示し、プロンプトエンジニアリングの先にある概念として位置づける。

**図表**:
- なし（2.5の戦略マトリクスを参照しつつ解説）

**キーポイント**:
- コンテキストエンジニアリングは「プロンプトエンジニアリング」の上位概念
- コンテキストの構成要素: システムプロンプト、会話履歴、ツール定義、外部情報
- 「何を入れるか」だけでなく「何を捨てるか」「いつ捨てるか」が重要
- コンテキストの質がエージェントの判断精度を直接左右する

## 章末の理解度チェック問題

1. Function Callingにおいて、LLMが実際に行うことと、ランタイムが行うことをそれぞれ説明せよ。
2. MCPの3つのプリミティブ（Tools、Resources、Prompts）の役割の違いを述べよ。
3. MCPホスト・MCPクライアント・MCPサーバーの関係を説明せよ。
4. コンテキストウィンドウを消費する4つの要素を挙げ、それぞれの特徴を述べよ。
