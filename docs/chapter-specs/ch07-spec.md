# 第7章 マルチエージェントの設計原則 ― 仕様書

## 章の概要

**章の目的**: マルチエージェントシステムの設計時に持つべき判断軸を学ぶ。第II部の締めくくりとして、理論を設計判断に落とし込む。

**目標ページ数**: 16p

**所属する部**: 第II部 マルチエージェントの理論

この章を読むために必要な前提:
- 第5章の内容（エージェント間通信の設計）
- 第6章の内容（状態管理とメモリアーキテクチャ）

## 節の構成

### 7.1 単一責任の原則 ― 1エージェント1責務

**概要**: ソフトウェア設計の単一責任の原則をエージェント設計に適用する。1つのエージェントが1つの明確な責務を持つことの利点と実践方法を解説する。

**図表**:
- 図7.1: 単一責任のエージェント設計例（責務が明確な設計 vs 曖昧な設計）

**キーポイント**:
- 責務が明確だとテスト・デバッグ・改善が容易
- 責務の境界をどこに引くかが設計力

### 7.2 エージェントの粒度設計 ― 細かすぎず粗すぎず

**概要**: エージェントをどの粒度で分割するかの設計判断を解説する。細かすぎると通信オーバーヘッドが増え、粗すぎるとシングルエージェントの問題に戻る。

**図表**:
- 図7.2: エージェント粒度のスペクトラム図

**キーポイント**:
- 粒度は一意に決まらない、トレードオフの判断
- 通信コスト vs 専門性のバランス

### 7.3 人間のフィードバックをどこに入れるか ― Human-in-the-Loop

**概要**: エージェントの処理フローにおいて、人間の判断・承認をどこに挿入するかの設計を解説する。安全性とユーザー体験のバランスを扱う。

**図表**:
- 図7.3: Human-in-the-Loopの挿入ポイント図

**キーポイント**:
- 重要な意思決定ポイントに人間を介入させる
- 介入が多すぎると自動化の意味がなくなる

### 7.4 フォールトトレランス ― エージェントが失敗したときの設計

**概要**: エージェントの失敗に対する耐性の設計を解説する。リトライ、フォールバック、グレースフルデグラデーションのパターンを示す。

**図表**:
- 図7.4: フォールトトレランスのパターン図（リトライ、フォールバック、デグラデーション）

**キーポイント**:
- 「失敗しない」ではなく「失敗しても動く」設計
- LLMの応答は非決定的であるため、失敗は日常

### 7.5 冪等性と再実行可能性 ― インフラ操作特有の要件

**概要**: インフラ操作を行うエージェントに特に重要な冪等性の設計を解説する。同じ操作を複数回実行しても安全であることの保証方法を示す。

**図表**:
- 図7.5: 冪等性のある操作とない操作の比較図

**キーポイント**:
- インフラ操作では冪等性が安全性の基盤
- Terraformの宣言的アプローチは冪等性の良い例

### 7.6 セキュリティ ― 最小権限、認証・認可の委譲

**概要**: マルチエージェントシステムにおけるセキュリティ設計を解説する。エージェントごとの最小権限、認証・認可の委譲パターンを扱う。

**図表**:
- 図7.6: エージェントごとの権限設計図

**キーポイント**:
- 各エージェントに必要最小限の権限のみを付与
- 認証情報の管理はVault等の専用サービスに委任

### 7.7 いつマルチエージェントにすべきか ― 複雑さのトレードオフ

**概要**: マルチエージェントの導入判断基準を解説する。シングルエージェントで十分なケースと、マルチが必要なケースの判断フレームワークを提供する。

**図表**:
- 表7.1: マルチエージェント導入の判断フレームワーク

**キーポイント**:
- マルチエージェントは複雑さを導入する
- シンプルに始めて、必要に応じてマルチにスケールする

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 4問
- 問題の種類: 概念の確認、判断問題、設計問題

### 問題案
1. エージェントの粒度設計における「細かすぎる」場合と「粗すぎる」場合のそれぞれの問題を述べよ（種類: 概念の確認）
2. Human-in-the-Loopを入れるべきポイントの判断基準を3つ挙げよ（種類: 概念の確認）
3. 次のシステムにおいて、マルチエージェントにすべきかシングルで十分か判断し、理由を述べよ（種類: 判断問題）
4. 3つのエージェント（ヒアリング、設計、デプロイ）に対する権限設計を行え（種類: 設計問題）
