# 第7章 マルチエージェントの設計原則 ― 仕様書

## 章の概要

**章の目的**: マルチエージェントシステムを設計する際に従うべき原則を体系的に整理し、読者が堅牢で保守性の高いシステムを設計するための判断基準を身につけられるようにする。第II部の締めくくりとして、理論編の知識を設計判断に昇華させる。
**目標ページ数**: 16p
**所属する部**: 第II部 マルチエージェントの理論

## 前提知識

本章を読む前に、読者が理解している必要がある概念:

- **第4章**: 6つの協調パターンとその使い分け
- **第5章**: エージェント間通信の方式（MCP、A2A、同期/非同期）
- **第6章**: 状態管理のパターン（共有メモリ、メッセージパッシング、永続化戦略）

## 節の構成

### 7.1 単一責任の原則

**概要**: マイクロサービスにおける単一責任の原則を、エージェント設計に適用する方法を扱う。一つのエージェントが一つの明確な役割を持つべきである理由と、責任の境界をどのように引くかの指針を示す。

**図表**:
- 図7.1: 責任の分離 ― 良い例と悪い例の対比図

**キーポイント**:
- 「一つのエージェント＝一つの専門性」が基本原則
- 責任の境界は「判断に必要な知識の範囲」で引く
- 複数の専門知識を一つのエージェントに詰め込むと、プロンプトが肥大化しLLMの精度が低下する
- マイクロサービスの原則との類似点と相違点

### 7.2 エージェントの粒度設計

**概要**: エージェントをどの粒度で分割するかの設計判断を扱う。粒度が細かすぎると通信オーバーヘッドが増大し、粗すぎると単一エージェントの問題に戻る。適切な粒度を見極めるための判断基準を示す。

**図表**:
- 図7.2: エージェント粒度のスペクトラム図（モノリシック ←→ マイクロ）
- 表7.1: 粒度設計のチェックリスト

**キーポイント**:
- 粒度のスペクトラム: モノリシック（単一エージェント）からマイクロ（機能単位のエージェント）まで
- 分割の判断基準: コンテキストウィンドウの制約、専門知識の独立性、並列実行の可能性
- 通信コストと分割メリットのトレードオフ
- 「まず粗く始めて、必要に応じて分割する」アプローチの推奨

### 7.3 Human-in-the-Loopをどこに入れるか

**概要**: エージェントの自律的な処理フローの中に、人間の判断・承認をどこに組み込むかの設計を扱う。リスクの大きさに応じた承認ポイントの設計、非同期承認フローの実現方法を整理する。

**図表**:
- 図7.3: Human-in-the-Loopの挿入ポイント図（リスクレベルに応じた配置）
- 表7.2: 操作リスクレベルと承認要否の判断マトリクス

**キーポイント**:
- 承認が必要な操作の基準: 不可逆性、影響範囲、コスト（課金額）
- 承認ポイントの配置: タスク開始前、重要な中間ステップ、最終実行前
- 非同期承認フロー: 人間の応答を待つ間のタスクの一時停止と再開
- 過剰な承認ポイントは生産性を損なう。リスクベースで必要最小限に設計する

### 7.4 フォールトトレランス

**概要**: エージェントの失敗を前提とした設計を扱う。LLMの応答品質のばらつき、ツール呼び出しの失敗、エージェント間通信の障害など、想定すべき故障モードとその対処パターンを整理する。

**図表**:
- 図7.4: フォールトトレランスパターンの概要図（リトライ / フォールバック / サーキットブレーカー）

**キーポイント**:
- エージェント固有の故障モード: LLMの幻覚、ツール呼び出し失敗、タイムアウト、コンテキスト超過
- リトライ戦略: 指数バックオフ、リトライ上限、リトライ可能な操作の識別
- フォールバック: 代替エージェント・代替手段への切り替え
- サーキットブレーカー: 連続失敗時に呼び出しを一時停止し、システム全体の破綻を防ぐ
- グレースフルデグラデーション: 一部機能が利用不可でも、縮退した形で処理を継続

### 7.5 冪等性と再実行可能性

**概要**: エージェントの操作が冪等であることの重要性と、タスク全体を安全に再実行できる設計を扱う。特にインフラ操作（OCI上のリソース作成・変更等）を行うエージェントにおいて不可欠な概念である。

**図表**:
- 表7.3: 操作の冪等性分類と設計指針

**キーポイント**:
- 冪等性: 同じ操作を複数回実行しても結果が変わらない性質
- インフラ操作における冪等性: Terraformの宣言的モデルがエージェント設計にも有用
- 再実行可能性: タスクが途中で失敗した場合に、最初からまたはチェックポイントから安全に再開できること
- 第6章の永続化戦略（チェックポイント、イベントソーシング）との関連

### 7.6 セキュリティ

**概要**: マルチエージェントシステム固有のセキュリティ課題と対策を扱う。エージェントの権限管理、プロンプトインジェクション対策、認証・認可の設計を整理する。

**図表**:
- 図7.5: エージェントの権限管理モデル（最小権限の原則の適用）

**キーポイント**:
- 最小権限の原則: 各エージェントには必要最小限のツール・データアクセス権限のみを付与
- プロンプトインジェクション対策: 外部入力がエージェントの動作を意図せず変更するリスクへの対処
- 認証・認可の分離: エージェント自身の認証と、エージェントが代行する操作の認可を分離して設計
- エージェント間通信の信頼モデル: どのエージェントを信頼するか、信頼の範囲をどう制限するか

### 7.7 いつマルチエージェントにすべきか

**概要**: 第II部の締めくくりとして、「そもそもマルチエージェントにすべきか」の判断基準を改めて整理する。第3章で提起した「なぜマルチが必要か」に対して、第II部で学んだ知識を踏まえた実践的な判断フレームワークを提示する。

**図表**:
- 図7.6: マルチエージェント採用の判断フローチャート

**キーポイント**:
- シングルエージェントで十分なケース: タスクが単純、専門知識が単一領域、低レイテンシ要件
- マルチエージェントが有効なケース: 複数領域の専門知識が必要、並列処理が可能、検証・承認フローがある
- マルチエージェントのコスト: 通信オーバーヘッド、状態管理の複雑性、デバッグの困難さ
- 「マルチエージェントにすべきか」の判断チェックリスト
- 第3章の問題提起に対する回答として、本節は第II部全体のまとめとなる

## 章末の理解度チェック問題

1. 一つのエージェントに複数の責任を持たせた場合に生じる問題を、コンテキストウィンドウの観点から説明せよ。
2. エージェントの粒度を「粗すぎる」「細かすぎる」それぞれの場合のデメリットを挙げ、適切な粒度を判断する基準を述べよ。
3. OCI上のリソース削除操作をエージェントが実行する場合、Human-in-the-Loopをどのように設計するか。リスク評価と合わせて説明せよ。
4. エージェントがLLMの幻覚により誤ったツール呼び出しを行った場合のフォールトトレランス設計を提案せよ。
5. あなたが設計するシステムにおいて、マルチエージェントにすべきかシングルエージェントで十分か、判断フローチャートを使って検討するシナリオを一つ挙げて説明せよ。
