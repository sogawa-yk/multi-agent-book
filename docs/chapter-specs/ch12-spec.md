# 第12章 ケーススタディ ― OKEクラスタの自動構築システム ― 仕様書

## 章の概要

**章の目的**: 第I部から第III部までに学んだ知識を総動員し、OKEクラスタの自動構築をテーマとした本格的なマルチエージェントシステムの設計を通じて、協調パターンの選定・エージェント分割・状態管理・エラーリカバリ・設計判断のプロセスを追体験させる。
**目標ページ数**: 24p
**所属する部**: 第III部 OCI上での実践（統合）

## 前提知識

- 第4章で扱う協調パターン（特にオーケストレーター型、直列パイプライン）
- 第5章で扱うエージェント間通信（MCP、A2A）
- 第6章で扱う状態管理とメモリアーキテクチャ
- 第7章で扱うマルチエージェントの設計原則（冪等性、Human-in-the-Loop等）
- 第10章で扱うOCI上でのマルチエージェントシステム構築の基礎
- 第11章で扱うOCIサービス群のエージェント活用パターン（Resource Manager、OKE、Vault等）
- Kubernetesの基本概念（クラスタ、ノード、Pod、Serviceの概要）

## 節の構成

### 12.1 要件と全体設計

**概要**: ケーススタディの要件を定義する。「ユーザーが自然言語で要件を伝えると、マルチエージェントシステムがOKEクラスタを自動構築する」というシナリオの全体像を示す。ユーザーのインプット例、期待されるアウトプット、システムの非機能要件（冪等性、リカバリ、監査ログ）を明確にする。協調パターンとしてオーケストレーター型を採用する理由を、第4章のパターン選定ガイドに基づいて説明する。

**図表**:
- 図12.1: OKEクラスタ自動構築システムの全体アーキテクチャ図

**キーポイント**:
- 自然言語による要件入力からインフラ構築完了までのエンドツーエンドフロー
- オーケストレーター型パターンの採用理由（タスクの動的な分解が必要、段階的な判断が求められる）
- 非機能要件の定義（冪等性、リカバリ可能性、監査証跡、Human-in-the-Loop）
- 各エージェントの責務分割の設計指針

### 12.2 ヒアリングエージェント

**概要**: ユーザーとの対話を通じて、OKEクラスタの要件を構造化するエージェントの設計を解説する。自然言語の曖昧な要件を、環境（本番/開発/検証）、ノード構成、ネットワーク要件、セキュリティ要件等の構造化されたスペックに変換する役割を担う。不足情報の検出と追加質問の生成ロジックを示す。

**図表**:
- 図12.2: ヒアリングエージェントの入出力と対話フロー

**キーポイント**:
- 自然言語の曖昧さを解消するための段階的なヒアリング戦略
- 出力スキーマの定義（構造化された要件仕様書）
- デフォルト値の推論と確認（例: 環境が未指定なら「開発環境」を提案）
- ヒアリング完了判定の基準（必須項目の充足チェック）

### 12.3 アーキテクチャエージェント

**概要**: ヒアリングエージェントの出力を受け取り、OKEクラスタのアーキテクチャ設計を行うエージェントの設計を解説する。VCN設計、サブネット構成、ノードプール設計、ネットワークポリシーの決定を行い、アーキテクチャドキュメントを生成する。OCI固有のベストプラクティスを設計に反映する仕組みを示す。

**図表**:
- 図12.3: アーキテクチャエージェントが生成するOKEクラスタ構成の例

**キーポイント**:
- 要件仕様書からアーキテクチャ設計への変換ロジック
- OCIのベストプラクティス（可用性ドメイン分散、セキュリティリスト設計等）の組み込み方
- 設計制約のバリデーション（リソースクォータ、リージョン制約等）
- アーキテクチャ決定の根拠を出力に含める設計（説明可能性）

### 12.4 IaC生成エージェント

**概要**: アーキテクチャ設計をTerraform HCLコードに変換するエージェントの設計を解説する。テンプレートベースの生成とLLMによる動的な調整の組み合わせ、生成コードのバリデーション（terraform validate相当）、モジュール構造の設計を示す。

**図表**:
- なし（12.1の全体アーキテクチャ図で位置づけを示す）

**キーポイント**:
- テンプレートとLLM生成のハイブリッドアプローチ（信頼性と柔軟性の両立）
- 生成コードの自己検証（構文チェック、必須リソースの存在確認）
- Terraformモジュールの分割設計（ネットワーク、クラスタ、ノードプール等）
- バージョン管理との連携（生成コードのGitコミット）

### 12.5 デプロイエージェント

**概要**: 生成されたTerraform HCLをOCI Resource Managerに投入し、実際にインフラをデプロイするエージェントの設計を解説する。スタック作成、プラン実行、プラン結果の確認（Human-in-the-Loop）、適用実行、完了確認までのフローを示す。非同期ジョブの監視とタイムアウト処理を含む。

**図表**:
- 図12.4: デプロイエージェントのワークフローとHuman-in-the-Loopポイント

**キーポイント**:
- Resource ManagerのAPIを介したスタック操作の設計
- プラン結果の要約（何が作成されるかを人間が判断しやすい形式で提示）
- Human-in-the-Loopによる適用前の承認フロー
- 非同期ジョブのポーリングとタイムアウト設計

### 12.6 検証エージェント

**概要**: デプロイ完了後のOKEクラスタが要件を満たしているかを検証するエージェントの設計を解説する。クラスタのヘルスチェック、ノード数・スペックの確認、ネットワーク疎通確認、Kubernetes APIの応答確認を行う。検証結果をレポートとして出力し、問題があればリカバリフローへの移行を判断する。

**図表**:
- 図12.5: 検証エージェントのチェック項目とリカバリ判断フロー

**キーポイント**:
- 検証項目のチェックリスト設計（宣言的な期待状態との比較）
- 検証結果のレポート生成（成功/失敗/警告の分類）
- 部分的な失敗の判断基準（致命的エラー vs 許容可能な警告）
- リカバリが必要な場合のオーケストレーターへのフィードバック

### 12.7 オーケストレーターの設計

**概要**: 上記5つのエージェントを統括するオーケストレーターの設計を解説する。タスクの分解と各エージェントへの委任、エージェント間のデータフロー管理、全体の状態管理、進捗の可視化を担う。オーケストレーターが「何を知っているべきか」と「何を知らなくてよいか」の境界線を設計する。

**図表**:
- 図12.6: オーケストレーターと各エージェント間の全体シーケンス図

**キーポイント**:
- オーケストレーターの責務の範囲（タスク分解・委任・結果統合・エラー判断）
- 各エージェント間のインターフェース設計（入力スキーマ・出力スキーマの定義）
- 全体の状態遷移管理（どのフェーズにいるか、次に何をすべきか）
- オーケストレーターのプロンプト設計（各エージェントの能力の記述、判断基準の明示）

### 12.8 失敗時のリカバリ

**概要**: マルチエージェントシステムにおける失敗シナリオとリカバリ戦略を解説する。エージェントレベルの失敗（LLMのハルシネーション、ツール呼び出しエラー）、インフラレベルの失敗（Resource Managerジョブの失敗、リソースクォータ超過）、システムレベルの失敗（オーケストレーターの異常停止）の各レイヤーでのリカバリパターンを示す。

**図表**:
- 図12.7: 失敗レイヤーごとのリカバリ戦略マッピング

**キーポイント**:
- 失敗の分類（一時的エラー vs 永続的エラー、リトライ可能 vs 人間介入必要）
- リトライ戦略（指数バックオフ、最大リトライ回数、別エージェントへのフォールバック）
- 部分的成功からの再開（チェックポイントと状態復元）
- terraform destroyによるクリーンアップ（冪等な後片付け）

### 12.9 設計判断の振り返り

**概要**: 本ケーススタディで行った設計判断を振り返り、「なぜそう設計したか」「別の選択肢はなかったか」を議論する。パターン選定、エージェント粒度、インターフェース設計、リカバリ戦略の各判断について、トレードオフを明示する。読者が自身のプロジェクトに知見を転用できるよう、設計判断のフレームワークを提供する。

**図表**:
- 表12.1: 設計判断のトレードオフ一覧

**キーポイント**:
- オーケストレーター型 vs 直列パイプラインの判断理由と、異なるケースでの選択基準
- エージェント粒度の判断（5つのエージェントに分割した理由、統合・分割の基準）
- 「ここはエージェントに任せるべきか、決定的なコードで書くべきか」の境界線
- 本ケーススタディの設計を他のユースケースに応用する際の考え方

## 章末の理解度チェック問題

1. 本ケーススタディでオーケストレーター型パターンを採用した理由を、直列パイプラインと比較して説明せよ。
2. ヒアリングエージェントが「ヒアリング完了」と判断する基準として、どのような設計が考えられるか述べよ。
3. IaC生成エージェントがテンプレートベースの生成とLLM生成を組み合わせるハイブリッドアプローチを採用する理由を説明せよ。
4. デプロイエージェントにおけるHuman-in-the-Loopの設計で、何をどのタイミングで人間に提示すべきか説明せよ。
5. 失敗時のリカバリにおいて、「一時的エラー」と「永続的エラー」の区別が重要な理由を述べよ。

## 章間の接続

- 第I部〜第III部の知識を総動員する統合章として位置づけられる

- 本章末で「ケーススタディを通じてマルチエージェントシステムの設計と構築を一通り体験した。次章では、こうしたシステムをどのようにテスト・デバッグするかを体系的に学ぶ」と接続する
