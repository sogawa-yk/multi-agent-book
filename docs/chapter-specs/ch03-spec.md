# 第3章 シングルエージェントの限界 ― なぜマルチが必要になるのか ― 仕様書

## 章の概要

**章の目的**: シングルエージェントが構造的に抱える5つの限界を体系的に分析し、具体例（OKEクラスタ構築）で限界を実感させた上で、マルチエージェントが提供する解決策の方向性を示す。第I部の結論であり、第II部への橋渡しとなる章である。
**目標ページ数**: 16p
**所属する部**: 第I部 シングルエージェントの深い理解

## 前提知識

- 第1章の内容（エージェントの定義、構成要素、ReActパターン）
- 第2章の内容（Function Calling、MCP、コンテキストウィンドウ、コンテキストエンジニアリング）
- OCIの基本的なサービス（コンピュート、ネットワーク、IAM等）の理解

## 節の構成

### 3.1 タスクの複雑性とコンテキスト爆発

**概要**: タスクが複雑になるにつれて、コンテキストウィンドウが急速に消費される問題（コンテキスト爆発）を分析する。第2章で解説したコンテキスト管理の戦略をもってしても、タスクの複雑性に対してシングルエージェントのコンテキストは構造的に不足することを示す。

**図表**:
- 図3.1: タスクの複雑性とコンテキスト消費の関係図

**キーポイント**:
- タスクの複雑性が増すと、必要なコンテキスト量は線形ではなく指数的に増加する
- ツール定義、会話履歴、中間結果がコンテキストを圧迫する
- コンテキストの圧縮・要約は情報の損失を伴う
- コンテキスト爆発はシングルエージェントの最も根本的な制約

### 3.2 専門性のジレンマ

**概要**: 1つのエージェントに多くの役割を持たせると、各役割の専門性が薄まる問題を分析する。「何でもできるエージェント」は「何も上手くできないエージェント」になるジレンマを示す。ツール数の増加によるLLMの選択精度低下（第2章で伏線）を定量的に説明する。

**図表**:
- なし（3.1の図を参照しつつテキストで解説）

**キーポイント**:
- ツール数の増加はLLMの選択精度を低下させる
- システムプロンプトに多くの役割を記述すると、指示の優先順位が曖昧になる
- 専門性の高いタスクには、専門のコンテキスト（ツール定義、ドメイン知識）が必要
- 「T型エージェント」（広く浅く + 一つ深く）は現実的には困難

### 3.3 信頼性の壁

**概要**: シングルエージェントにおいて、エージェントの出力を検証する仕組みがない問題を分析する。自分で生成し自分で検証する「自己検証」の限界を示し、独立した検証者の必要性を論じる。

**図表**:
- 図3.2: シングルエージェントの失敗パターン図（自己検証の限界、エラー蓄積、回復不能）

**キーポイント**:
- 自己検証（自分の出力を自分で評価）にはバイアスがある
- エラーが蓄積し、後のステップで回復不能になるパターン
- Human-in-the-Loopは部分的な解決策だが、スケールしない
- 独立した「評価者」エージェントの必要性がここで示唆される

### 3.4 並行処理の不在

**概要**: シングルエージェントは本質的に逐次処理であり、独立したタスクを並行に実行できない問題を分析する。ReActループが直列であることの構造的制約を示し、並列処理の潜在的な効率性を論じる。

**図表**:
- なし（テキストで解説。図3.1の複雑性図を参照）

**キーポイント**:
- ReActループは本質的に逐次処理（1ステップずつ）
- 独立したサブタスクも直列に処理するため、実行時間が長くなる
- 並列Function Callingは部分的な解決策だが、タスクレベルの並行性は提供しない
- 実行時間の増加はコスト増加にも直結する

### 3.5 責任分界の曖昧さ

**概要**: シングルエージェントでは、タスクの計画・実行・検証・報告の全責任が1つのエージェントに集中し、問題発生時の原因特定や責任分界が困難になる問題を分析する。運用・デバッグの観点からの限界を示す。

**図表**:
- なし（テキストで解説）

**キーポイント**:
- 全責任が1つのエージェントに集中する「モノリシック」な構造
- 問題発生時に、計画の誤りか実行の誤りか判断が困難
- ログの分析が複雑化する（すべてが1つのコンテキストに混在）
- 責任の分離はソフトウェア設計の基本原則（単一責任の原則）

### 3.6 具体例: OKEクラスタ構築を1エージェントでやろうとすると

**概要**: OKEクラスタの自動構築という具体的なタスクを題材に、3.1〜3.5で分析した限界がどのように顕在化するかを示す。ネットワーク設計、Terraform生成、セキュリティ設定、検証といった多岐にわたるサブタスクを1つのエージェントで処理しようとした場合の問題を具体的に描写する。

**図表**:
- なし（テキストで具体的に描写。第12章のケーススタディの伏線）

**キーポイント**:
- OKEクラスタ構築は複数の専門領域（ネットワーク、コンピュート、セキュリティ、Kubernetes）にまたがる
- 必要なツール数が20を超え、LLMの選択精度が低下する
- Terraform生成と検証を同一コンテキストで行うとコンテキストが爆発する
- この例が第12章のケーススタディの導入となる

### 3.7 マルチエージェントが約束するもの

**概要**: 3.1〜3.6で分析した各限界に対して、マルチエージェントがどのような解決策を提供するかを概観する。詳細は第II部で扱うが、ここでは「なぜマルチエージェントなのか」の答えを提示し、読者の動機づけを完了する。

**図表**:
- なし（テキストで概観。表形式で限界と解決策の対応を示す）

**キーポイント**:
- コンテキスト爆発 → コンテキストの分割（各エージェントが専門コンテキストを持つ）
- 専門性のジレンマ → 役割の分離（各エージェントが専門領域を担当）
- 信頼性の壁 → 独立した検証（評価者エージェントによるクロスチェック）
- 並行処理の不在 → タスクの並列実行（独立サブタスクを複数エージェントで同時処理）
- 責任分界の曖昧さ → 責任の明確化（各エージェントの責務が明確）

## 章末の理解度チェック問題

1. コンテキスト爆発が発生するメカニズムを、コンテキストウィンドウの消費要素を挙げて説明せよ。
2. ツール数の増加がエージェントの性能に与える影響を述べよ。
3. シングルエージェントにおける「自己検証の限界」とは何か。具体例を挙げて説明せよ。
4. OKEクラスタ構築のようなタスクにおいて、シングルエージェントが直面する限界を3つ挙げよ。
5. マルチエージェントが「コンテキスト爆発」と「専門性のジレンマ」をどのように解決するか、それぞれ説明せよ。
