# 第3章 シングルエージェントの限界 ― なぜマルチが必要になるのか ― 仕様書

## 章の概要

**章の目的**: 単一エージェントの限界を具体的に示し、マルチエージェントへの移行の動機を理解する。第I部の締めくくりとして、第II部以降への強い動機づけを行う。

**目標ページ数**: 16p

**所属する部**: 第I部 シングルエージェントの深い理解

## 前提知識

この章を読むために必要な前提:
- 第2章の内容（ツール、MCP、コンテキスト管理の仕組み）
- エージェントの構成要素と動作パターン（第1章）

## 節の構成

### 3.1 タスクの複雑性とコンテキスト爆発 ― 1つのエージェントに全部やらせるとどうなるか

**概要**: タスクが複雑になるにつれ、1つのエージェントのコンテキストウィンドウが爆発する問題を解説する。ツール定義・会話履歴・中間結果が蓄積し、性能が劣化するメカニズムを示す。

**図表**:
- 図3.1: タスク複雑性とコンテキスト使用量の関係図

**キーポイント**:
- コンテキストウィンドウの有限性がスケーラビリティを制限する
- 情報量の増加に伴い、LLMの注意力が分散し精度が低下する

### 3.2 専門性のジレンマ ― 汎用プロンプト vs 専門プロンプト

**概要**: 1つのエージェントに複数の専門性を持たせようとする際のジレンマを解説する。汎用的なプロンプトでは精度が下がり、専門的にすると対応範囲が狭くなる問題を示す。

**図表**:
- 図3.2: 汎用性と専門性のトレードオフ図

**キーポイント**:
- 「何でもできるエージェント」は「何も上手くできないエージェント」になりがち
- 人間の組織でも専門家を分けるのと同じ原理

### 3.3 信頼性の壁 ― 長いツールチェーンでのエラー伝播

**概要**: ツール呼び出しが連鎖する中でのエラー伝播問題を解説する。1つのエラーが後続の全処理に影響する構造的な脆弱性を示す。

**図表**:
- 図3.3: ツールチェーンにおけるエラー伝播の図

**キーポイント**:
- 信頼性は各ステップの成功率の積で決まる
- 長いチェーンほど全体の信頼性は指数的に低下する

### 3.4 並行処理の不在 ― 順次実行の非効率

**概要**: シングルエージェントが本質的に逐次処理であることの非効率さを解説する。独立したタスクを同時に処理できないことによる時間的な無駄を示す。

**図表**:
- 図3.4: 順次実行と並行実行の時間比較図

**キーポイント**:
- 独立したタスクは並列に実行できるはずだが、シングルでは不可能
- 実行時間はボトルネックに支配される

### 3.5 責任分界の曖昧さ ― 誰が何を決めたのかわからない問題

**概要**: 1つのエージェントが全てを処理する場合、意思決定の追跡が困難になる問題を解説する。デバッグ、監査、改善の障害となる。

**図表**:
- 図3.5: シングルエージェントの意思決定ブラックボックス図

**キーポイント**:
- 責任の所在が不明確だと改善が困難
- 監査やコンプライアンスの要件を満たせない

### 3.6 具体例：OKEクラスタ構築を1エージェントでやろうとすると

**概要**: OCIのOKEクラスタ構築という実務的なユースケースを例に、上記の限界が具体的にどう現れるかを示す。ヒアリング→設計→IaC生成→デプロイ→検証を1エージェントで行う困難さを体験的に理解する。

**図表**:
- 図3.6: OKEクラスタ構築の全工程と1エージェントでの処理の問題点

**キーポイント**:
- 実務的なユースケースで限界を実感する
- 第12章のケーススタディの伏線となる

### 3.7 マルチエージェントが約束するもの ― 分業、専門化、並行、検証

**概要**: シングルの限界に対するマルチエージェントの解決策を概観する。分業、専門化、並行処理、相互検証の4つの利点を提示し、第II部への橋渡しを行う。

**図表**:
- 表3.1: シングルエージェントの限界とマルチエージェントの解決策の対応表

**キーポイント**:
- マルチエージェントは銀の弾丸ではなく、複雑さとのトレードオフ
- 限界を正しく認識することが、適切な設計判断の出発点

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 4問
- 問題の種類: 概念の確認、判断問題

### 問題案
1. シングルエージェントの5つの限界を列挙し、それぞれを1文で説明せよ（種類: 概念の確認）
2. あるタスクにおいてシングルエージェントの信頼性が90%のステップを5つ連鎖させた場合、全体の成功率はいくらか（種類: 概念の確認）
3. 次のユースケースのうち、シングルエージェントで十分なものとマルチエージェントが必要なものを分類せよ（種類: 判断問題）
4. マルチエージェントが導入する「新たな複雑さ」にはどのようなものがあるか（種類: 概念の確認）
