# 第4章 マルチエージェントの協調パターン ― 仕様書

## 章の概要

**章の目的**: 協調パターンを体系的に理解し、ユースケースに応じた選択ができるようになる。本書の理論的核心となる章。

**目標ページ数**: 24p

**所属する部**: 第II部 マルチエージェントの理論

## 前提知識

この章を読むために必要な前提:
- 第3章の内容（シングルエージェントの限界と、マルチへの動機）
- エージェントの構成要素と動作パターン（第1章）

## 節の構成

### 4.1 パターン分類の軸 ― 制御の集中度 × 通信の方向性

**概要**: マルチエージェントの協調パターンを分類するための2つの軸（制御の集中度と通信の方向性）を導入する。この軸を使って以降のパターンを体系的に位置づける。

**図表**:
- 図4.1: パターン分類のマトリクス（制御の集中度 × 通信の方向性）

**キーポイント**:
- 分類軸を持つことで、パターンの比較と選択が容易になる
- 制御の集中度：中央集権 ↔ 分散自律
- 通信の方向性：一方向 ↔ 双方向

### 4.2 直列パイプライン（Sequential）

**概要**: エージェントが順番に処理を行い、前のエージェントの出力を次の入力とするパターンを解説する。段階的な変換や承認フローに適する。

**図表**:
- 図4.2: 直列パイプラインのアーキテクチャ図
- OCI例: ヒアリング → 設計 → Terraform生成 → Resource Managerデプロイ

**キーポイント**:
- 最もシンプルなマルチエージェントパターン
- 各ステップの入出力が明確に定義できる場合に有効

### 4.3 並列ファンアウト/ファンイン（Parallel）

**概要**: 複数のエージェントに同時にタスクを分配し、結果を集約するパターンを解説する。独立した調査や同時検証に適する。

**図表**:
- 図4.3: 並列ファンアウト/ファンインのアーキテクチャ図
- OCI例: 複数リージョンの同時ヘルスチェック

**キーポイント**:
- タスクが独立していることが前提条件
- 集約ロジックの設計が品質を左右する

### 4.4 オーケストレーター型（Orchestrator）

**概要**: 中央のオーケストレーターエージェントがタスクを動的に分解し、サブエージェントに分配・統合するパターンを解説する。

**図表**:
- 図4.4: オーケストレーター型のアーキテクチャ図
- OCI例: インフラ構築の全体制御

**キーポイント**:
- タスク分解が動的で、事前に全ステップが確定しないケースに適する
- オーケストレーター自体がLLMベースの場合、柔軟だが予測が難しい

### 4.5 スーパーバイザー型（Supervisor / Hierarchical）

**概要**: 階層的な監督構造で上位エージェントが下位に委任し、結果を監督するパターンを解説する。オーケストレーターとの違いを明確にする。

**図表**:
- 図4.5: スーパーバイザー型のアーキテクチャ図

**キーポイント**:
- オーケストレーター：タスク分解と統合、スーパーバイザー：監督と委任
- 階層構造により、大規模なシステムにスケールできる

### 4.6 コレオグラフィ型（Choreography / Peer-to-Peer）

**概要**: 中央制御なしにエージェントが自律的に協調するパターンを解説する。A2Aプロトコルとの関係にも触れる。

**図表**:
- 図4.6: コレオグラフィ型のアーキテクチャ図

**キーポイント**:
- 単一障害点がないが、全体の挙動が予測しにくい
- ルールベースの協調プロトコルが必要

### 4.7 評価者ループ（Evaluator / Critic）

**概要**: 生成エージェントと評価エージェントが生成→評価→修正のサイクルを繰り返すパターンを解説する。品質保証パターンとして活用する。

**図表**:
- 図4.7: 評価者ループのフロー図

**キーポイント**:
- 品質の「自己改善」を実現する
- 収束条件の設計が重要（無限ループの回避）

### 4.8 パターンの組み合わせ ― 実システムはハイブリッド

**概要**: 実際のシステムでは複数のパターンを組み合わせることを示す。ハイブリッド構成の例と設計上の注意点を解説する。

**図表**:
- 図4.8: ハイブリッドパターンの例（オーケストレーター + 評価者ループ）

**キーポイント**:
- 実システムは「純粋な1パターン」であることは稀
- パターンの組み合わせ方にも設計判断がある

### 4.9 パターン選定ガイド ― ディシジョンツリーとトレードオフ表

**概要**: ユースケースに応じてどのパターンを選択すべきかの判断基準を提供する。ディシジョンツリーとトレードオフ比較表を示す。

**図表**:
- 図4.9: パターン選定ディシジョンツリー
- 表4.1: 協調パターンのトレードオフ比較表

**キーポイント**:
- パターン選定は要件に基づく設計判断
- トレードオフを理解した上で選択する

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 5問
- 問題の種類: 概念の確認、判断問題、設計問題

### 問題案
1. 6つの協調パターンを列挙し、それぞれの特徴を1文で説明せよ（種類: 概念の確認）
2. オーケストレーター型とスーパーバイザー型の違いを説明せよ（種類: 概念の確認）
3. 次のユースケースに最も適する協調パターンを選び、理由を述べよ：「複数のクラウドリージョンのセキュリティ設定を同時に監査する」（種類: 判断問題）
4. 「ドキュメントの自動生成→レビュー→修正」のフローに適するパターンの組み合わせを設計せよ（種類: 設計問題）
5. コレオグラフィ型を採用すべきケースと避けるべきケースをそれぞれ挙げよ（種類: 判断問題）
