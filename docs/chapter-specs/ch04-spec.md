# 第4章 マルチエージェントの協調パターン ― 仕様書

## 章の概要

**章の目的**: マルチエージェントシステムにおける6つの主要な協調パターン（直列パイプライン、並列ファンアウト/ファンイン、オーケストレーター型、スーパーバイザー型、コレオグラフィ型、評価者ループ）を体系的に解説し、ユースケースに応じたパターン選定の判断力を読者に提供する。本書の理論的な核心となる章である。
**目標ページ数**: 24p
**所属する部**: 第II部 マルチエージェントの理論

## 前提知識

- 第1章〜第3章の内容（エージェントの定義、構成要素、ツール、コンテキスト、シングルエージェントの限界）
- 特に第3章の5つの限界（コンテキスト爆発、専門性のジレンマ、信頼性の壁、並行処理の不在、責任分界の曖昧さ）

## 節の構成

### 4.1 パターン分類の軸

**概要**: 6つの協調パターンを分類する3つの軸（制御方式: 集中型/分散型、データフロー: 直列/並列/動的、エージェント関係: 対等/階層）を提示する。各パターンをこれらの軸で位置づけることで、パターン間の違いと関係性を構造的に理解できるようにする。

**図表**:
- なし（テキストで3つの軸を説明。4.9節のディシジョンツリーで統合）

**キーポイント**:
- 制御方式: 集中型（オーケストレーター、スーパーバイザー）vs 分散型（コレオグラフィ）
- データフロー: 直列（パイプライン）vs 並列（ファンアウト/ファンイン）vs 動的（オーケストレーター）
- エージェント関係: 対等（コレオグラフィ、ファンアウト/ファンイン）vs 階層（スーパーバイザー）
- これらの軸が「なぜこのパターンを選ぶか」の判断基準となる

### 4.2 直列パイプライン

**概要**: エージェントが直列に接続され、前のエージェントの出力が次のエージェントの入力となるパターンを解説する。段階的な変換、品質向上パイプライン、承認フロー等のユースケースを示す。最もシンプルなマルチエージェントパターンとして、他のパターンとの比較基準となる。

**図表**:
- 図4.1: 直列パイプラインのアーキテクチャ図

**キーポイント**:
- 前のエージェントの出力が次のエージェントの入力（データの一方向の流れ）
- 各エージェントが独立したコンテキストを持つため、コンテキスト爆発を緩和
- 適用例: 文書生成→レビュー→修正、要件分析→設計→実装
- トレードオフ: シンプルだが柔軟性に欠ける、エラー伝播のリスク

### 4.3 並列ファンアウト/ファンイン

**概要**: タスクを複数のエージェントに同時に分配（ファンアウト）し、結果を集約（ファンイン）するパターンを解説する。独立した調査の並列実行、複数観点からの同時検証等のユースケースを示す。第3章で指摘した「並行処理の不在」を直接解決するパターンである。

**図表**:
- 図4.2: 並列ファンアウト/ファンインのアーキテクチャ図

**キーポイント**:
- タスクの分配（ファンアウト）と結果の集約（ファンイン）の2つのフェーズ
- 独立したサブタスクが前提（サブタスク間に依存関係がないこと）
- ファンインの設計（結果の統合方法: マージ、投票、選択）が品質を左右する
- 適用例: 複数ソースからの情報収集、複数観点からのコードレビュー

### 4.4 オーケストレーター型

**概要**: 中央のオーケストレーターエージェントが動的にタスクを分解・分配し、サブエージェントの結果を統合するパターンを解説する。最も柔軟なパターンであり、タスクの複雑さに応じて実行計画を動的に変更できる。Claude Codeのサブエージェント機能がこのパターンの典型例である。

**図表**:
- 図4.3: オーケストレーター型のアーキテクチャ図

**キーポイント**:
- オーケストレーターが「計画」と「統合」を担い、サブエージェントが「実行」を担う
- 実行計画は動的に変更可能（前のサブタスクの結果に基づいて次を決定）
- 最も柔軟だが、オーケストレーター自身のコンテキスト管理が課題
- 適用例: 複雑なタスクの動的分解、多段階の調査・分析

### 4.5 スーパーバイザー型

**概要**: 階層的な監督構造を持つパターンを解説する。オーケストレーター型とは異なり、監督と委任の関係性を重視し、エスカレーション（問題の上位への報告）やフォールバック（代替エージェントへの切り替え）の仕組みを持つ。組織的なワークフローのモデリングに適する。

**図表**:
- 図4.4: スーパーバイザー型のアーキテクチャ図

**キーポイント**:
- オーケストレーターとの違い: スーパーバイザーは「監督」の役割、エスカレーション機構を持つ
- ワーカーエージェントが自律的にタスクを遂行し、スーパーバイザーに報告する
- エスカレーション: ワーカーが解決できない問題をスーパーバイザーに上げる
- 適用例: 運用監視、承認フロー、品質管理

### 4.6 コレオグラフィ型

**概要**: 中央制御なしにエージェント同士が自律的に協調するパターンを解説する。各エージェントが独立した判断で他のエージェントと通信するピアツーピア型のアーキテクチャ。分散システムの知見が活かされるパターンであり、スケーラビリティに優れる。

**図表**:
- 図4.5: コレオグラフィ型のアーキテクチャ図

**キーポイント**:
- 中央のコントローラーが存在しない（各エージェントが自律的に判断）
- エージェント間の直接通信（ピアツーピア）
- スケーラビリティに優れるが、全体の振る舞いの予測・デバッグが困難
- 適用例: 分散型のイベント処理、マイクロサービス的なエージェント構成

### 4.7 評価者ループ

**概要**: 生成→評価→修正のサイクルを繰り返すパターンを解説する。あるエージェントが出力を生成し、別のエージェント（評価者/批評者）がそれを評価・修正指示する。第3章で指摘した「信頼性の壁」を直接解決するパターンである。

**図表**:
- 図4.6: 評価者ループのアーキテクチャ図

**キーポイント**:
- 「生成者」と「評価者」の役割分離による品質向上
- ループの終了条件の設計（品質基準到達、最大反復回数）が重要
- 評価者の独立性が品質に直結する（同じLLMでも異なるプロンプト・コンテキストで分離可能）
- 適用例: コード生成と品質検証、文書作成とレビュー、Terraformの計画と検証

### 4.8 パターンの組み合わせ

**概要**: 実際のシステムでは、6つのパターンを単独で使うことは稀であり、複数のパターンを組み合わせて使うことを解説する。オーケストレーター型の中でファンアウト/ファンインを使う、パイプラインの各段階に評価者ループを挟む、等の組み合わせパターンを示す。

**図表**:
- なし（テキストで組み合わせ例を描写。4.2〜4.7の図を参照）

**キーポイント**:
- 実システムでは複数パターンの組み合わせが一般的
- 組み合わせの典型例: オーケストレーター + ファンアウト/ファンイン、パイプライン + 評価者ループ
- 過度な複雑さは避ける（必要最小限のパターンを選択する）
- 第12章のOKEクラスタ構築ケーススタディで組み合わせの実例を示す

**概要**: 6つのパターンの特徴を比較し、ユースケースに応じたパターン選定のためのディシジョンツリーを提示する。「どのパターンを使うべきか」に対する実践的な判断フレームワークを提供する。

**図表**:
- 表4.1: 協調パターン比較表（6パターンの特性比較）
- 図4.7: パターン選定ディシジョンツリー

**キーポイント**:
- 比較軸: 制御方式、柔軟性、複雑性、デバッグ容易性、スケーラビリティ
- 判断の出発点は「タスクの性質」（独立性、順序依存性、品質要求）
- 「最もシンプルなパターンで十分なら、それを使う」が基本原則
- パターン選定は設計の初期段階で行うが、実装しながら調整する

## 章末の理解度チェック問題

1. 6つの協調パターンを分類する3つの軸を挙げ、それぞれの軸で各パターンがどこに位置するか述べよ。
2. オーケストレーター型とスーパーバイザー型の違いを、制御方式とエスカレーションの観点から説明せよ。
3. 評価者ループが第3章で指摘した「信頼性の壁」をどのように解決するか説明せよ。
4. 以下のユースケースにおいて、最も適切な協調パターンを選択し、その理由を述べよ: (a) 複数のセキュリティ観点からインフラ構成を同時に検証する (b) 文書を段階的に生成・レビュー・修正するワークフロー
5. パターンの組み合わせが有効な場面を1つ挙げ、どのパターンをどう組み合わせるか具体的に述べよ。

## 章間の接続

### 前の章（第3章: シングルエージェントの限界）との関係
- 第3章で分析した5つの限界に対する解決策として、6つの協調パターンを提示する
- コンテキスト爆発 → コンテキストの分割（各パターンで実現）
- 専門性のジレンマ → 役割の分離（各エージェントが専門領域を担当）
- 信頼性の壁 → 評価者ループ
- 並行処理の不在 → ファンアウト/ファンイン
- 責任分界の曖昧さ → 各パターンによる明確な責任分離

### 次の章（第5章: エージェント間の通信）への接続
- 本章で示した協調パターンを実現するためには、エージェント間の通信メカニズムが必要
- 特にコレオグラフィ型やオーケストレーター型では、エージェント間のメッセージ設計が重要
- 本章末で「協調パターンの構造を理解した。しかし、エージェント同士が実際にどのように通信するかをまだ議論していない。次章では、MCP、A2A等の通信プロトコルとメッセージ設計を掘り下げる」と接続する

### 第6章（状態管理とメモリアーキテクチャ）との関係
- 本章の協調パターンでは、エージェント間で状態を共有・管理する仕組みが必要
- 第6章でブラックボード、イベントソーシング等の状態管理パターンを扱う
