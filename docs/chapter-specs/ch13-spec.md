# 第13章 マルチエージェントのテストとデバッグ ― 仕様書

## 章の概要

**章の目的**: マルチエージェント特有のテスト・デバッグ手法を学ぶ。LLMの非決定性を踏まえた実践的なテスト戦略を理解する。

**目標ページ数**: 18p

**所属する部**: 第IV部 運用と発展

## 前提知識

この章を読むために必要な前提:
- 第10章の内容（マルチエージェントシステムの構築）
- 一般的なソフトウェアテストの基礎知識

## 節の構成

### 13.1 何をテストするのか ― 個別エージェント、連携、End-to-End

**概要**: マルチエージェントシステムにおけるテスト対象を3つのレベル（個別エージェント、エージェント間連携、End-to-End）に整理する。

**図表**:
- 図13.1: テスト戦略のピラミッド図（単体→統合→E2E）

**キーポイント**:
- テストレベルごとに目的とコストが異なる
- ピラミッドの下層（単体テスト）を厚くする

### 13.2 エージェント単体テスト ― LLMのモック、ツールのスタブ

**概要**: 個別エージェントのテスト方法を解説する。LLMの応答をモックし、ツールをスタブ化することで決定的なテストを実現する手法を示す。

**図表**:
- 図13.2: エージェント単体テストの構成図（モック/スタブ配置）

**キーポイント**:
- LLMのモックにより非決定性を排除
- ツール側のスタブで外部依存を排除

### 13.3 統合テスト ― エージェント間の連携テスト戦略

**概要**: 複数のエージェント間の連携をテストする戦略を解説する。境界値テスト、メッセージ形式の検証、エラー伝播のテストを示す。

**図表**:
- 図13.3: 統合テストの範囲と手法の図

**キーポイント**:
- インターフェースの契約テスト
- エラーケースの網羅が重要

### 13.4 End-to-Endテスト ― シナリオベースのテスト設計

**概要**: システム全体を対象としたE2Eテストの設計方法を解説する。シナリオベースのテスト設計とゴールデンパステストの手法を示す。

**図表**:
- 図13.4: E2Eテストのシナリオ設計例

**キーポイント**:
- E2Eテストはコストが高いため、重要なシナリオに絞る
- ゴールデンパス（正常系の主要フロー）のテストを優先

### 13.5 LLMの非決定性への対処 ― 評価基準の設計

**概要**: LLMの出力が毎回異なるという本質的な課題への対処法を解説する。完全一致ではなく、品質基準に基づく評価手法を示す。

**図表**:
- 表13.1: LLM出力の評価基準設計例

**キーポイント**:
- 「完全一致」ではなく「要件を満たすか」で評価する
- 評価関数の設計がテストの質を決める

### 13.6 トレーシングとデバッグ ― エージェントの思考過程を可視化する

**概要**: エージェントの思考過程（推論→行動→観察のログ）を可視化するトレーシング手法を解説する。問題発生時のデバッグワークフローを示す。

**図表**:
- 図13.5: トレーシングの可視化例

**キーポイント**:
- トレーシングはデバッグの最重要ツール
- 構造化されたログ出力が前提

### 13.7 回帰テスト ― モデル更新時の品質保証

**概要**: LLMモデルの更新（バージョンアップ、モデル変更）時の品質保証手法を解説する。ベンチマークテストセットの設計と回帰検知の仕組みを示す。

**図表**:
- 図13.6: 回帰テストのワークフロー図

**キーポイント**:
- モデル更新は予告なく品質に影響する可能性がある
- ベンチマークテストセットの継続的な実行

## 章末の理解度チェック問題

### 問題の方針
- 問題数: 4問
- 問題の種類: 概念の確認、判断問題、設計問題

### 問題案
1. テストピラミッドの3レベルにおいて、各レベルのテスト目的とコストの違いを述べよ（種類: 概念の確認）
2. LLMの出力が非決定的であることがテストに与える影響と、その対処法を述べよ（種類: 概念の確認）
3. あるエージェントが時々誤ったツールを選択する問題のデバッグ手順を述べよ（種類: 判断問題）
4. 3つのエージェントから成るシステムの統合テスト戦略を設計せよ（種類: 設計問題）
