# 第13章 マルチエージェントのテストとデバッグ ― 仕様書

## 章の概要

**章の目的**: マルチエージェントシステム特有のテスト課題（LLMの非決定性、エージェント間の相互作用、長い実行チェーン）を整理し、単体テスト・統合テスト・E2Eテストの各レベルでの実践的なテスト戦略と、トレーシングを活用したデバッグ手法を提供する。
**目標ページ数**: 18p
**所属する部**: 第IV部 運用と発展

## 前提知識

- 第10章で扱うマルチエージェントシステムの構築経験
- 第12章のケーススタディで体験したマルチエージェントの設計と実装
- ソフトウェアテストの基本概念（単体テスト、統合テスト、E2Eテストの区分）
- 第4章で扱う協調パターンの理解（テスト対象の構造を理解するため）

## 節の構成

### 13.1 何をテストするのか ― マルチエージェントのテスト課題

**概要**: マルチエージェントシステムのテストが従来のソフトウェアテストと異なる点を整理する。LLMの非決定性、エージェント間の創発的な振る舞い、長い実行チェーンの再現困難性、テストコストの高さ（LLM API呼び出し費用）といった課題を明示し、本章で扱うテスト戦略の全体像を示す。

**図表**:
- 図13.1: マルチエージェントのテスト戦略ピラミッド

**キーポイント**:
- 従来のソフトウェアテストとの違い（決定的 vs 非決定的、入力空間の広さ）
- テストピラミッド（単体→統合→E2E）のマルチエージェント版への適用
- テストコストの構造（LLM API呼び出し回数とテスト実行時間）
- 「何を保証すべきか」の優先順位付け（安全性 > 正確性 > 効率性）

### 13.2 エージェント単体テスト

**概要**: 個々のエージェントを分離してテストする手法を解説する。LLMをモック（スタブ）に置き換えた決定的テスト、ツール呼び出しの正当性検証、プロンプトの品質テスト（期待する構造化出力が得られるか）を示す。エージェントの入出力インターフェースに基づくコントラクトテストの概念も導入する。

**図表**:
- 図13.2: エージェント単体テストの構成（LLMモックとツールスタブ）

**キーポイント**:
- LLMモックによる決定的テスト（事前定義された応答を返すスタブ）
- ツール呼び出しの検証（正しいツールが正しい引数で呼ばれるか）
- プロンプトテスト（出力スキーマへの準拠、必須フィールドの存在確認）
- コントラクトテスト（エージェントの入出力インターフェースの契約遵守）

### 13.3 統合テスト

**概要**: 複数のエージェントを組み合わせた統合テストの手法を解説する。エージェント間のインターフェース整合性、データフローの正常性、エラー伝播の挙動を検証する。オーケストレーターと子エージェントの組み合わせテスト、直列パイプラインのステップ間テストなど、協調パターンごとのテストパターンを示す。

**図表**:
- なし（13.2の構成図を拡張して説明する）

**キーポイント**:
- エージェント間のインターフェース整合性テスト（出力スキーマと入力スキーマの一致）
- データフローテスト（エージェントAの出力がエージェントBで正しく処理されるか）
- エラー伝播テスト（子エージェントの失敗がオーケストレーターに正しく報告されるか）
- 協調パターンごとのテスト戦略の違い

### 13.4 End-to-Endテスト

**概要**: システム全体を通したE2Eテストの設計と実行方法を解説する。実際のLLMとツールを使った本番に近いテスト環境の構築、テストシナリオの設計、合格判定基準の設定方法を示す。テストコストの管理と、頻度の最適化（CI/CDとの統合）を含む。

**図表**:
- なし（13.1のテストピラミッド図で位置づけを示す）

**キーポイント**:
- テストシナリオの設計（ゴールデンパス + エッジケース + 失敗シナリオ）
- テスト環境の設計（本番環境との差異を最小化しつつコストを抑制）
- 合格判定基準の設定（厳密一致ではなく、品質基準ベースの判定）
- CI/CDパイプラインとの統合（どの頻度でE2Eテストを実行するか）

### 13.5 LLMの非決定性への対処

**概要**: LLMの出力が毎回異なるという根本的な課題に対する実践的な対処法を解説する。temperatureの制御、出力の構造化（JSON Schema等）による揺れの抑制、統計的なテスト（N回実行して成功率を評価）、アサーションの設計（完全一致ではなく意味的一致の判定）を示す。

**図表**:
- 図13.3: 非決定性の対処戦略マトリクス（抑制策 × 許容策）

**キーポイント**:
- 非決定性の抑制（temperature=0、構造化出力、Few-shotプロンプト）
- 非決定性の許容（N回実行、成功率ベースの判定、意味的一致のアサーション）
- 「出力の形式」と「出力の内容」を分離してテストする戦略
- シード値の固定による再現性確保（対応モデルの場合）

### 13.6 トレーシングとデバッグ

**概要**: マルチエージェントシステムのデバッグを支援するトレーシング手法を解説する。分散トレーシングの概念をマルチエージェントに適用し、エージェント間の呼び出しチェーンの可視化、各エージェントの思考過程のログ収集、ボトルネックの特定方法を示す。OCI Logging/Logging Analyticsを活用したトレース収集の実践例を含む。

**図表**:
- 図13.4: マルチエージェントのトレーシング可視化例

**キーポイント**:
- 分散トレーシングの概念（トレースID、スパン、親子関係）のマルチエージェントへの適用
- 構造化ログの設計（エージェントID、ステップ番号、ツール呼び出し、LLM入出力）
- トレースの可視化（タイムライン表示、コールグラフ）
- デバッグのワークフロー（問題の再現→トレースの収集→原因の特定→修正）

### 13.7 回帰テストとテスト資産の管理

**概要**: マルチエージェントシステムの回帰テスト戦略を解説する。プロンプト変更やモデル更新時の品質保証、テストケースの蓄積とメンテナンス、ゴールデンデータセットの管理方法を示す。テスト資産（テストケース、モック応答、期待出力）のバージョン管理と、テストの陳腐化への対処を含む。

**図表**:
- なし（13.1のテストピラミッド図を回帰テストの文脈で再参照する）

**キーポイント**:
- プロンプト変更時の影響範囲の特定と回帰テストの実行
- ゴールデンデータセット（入力→期待出力のペア集合）の設計と管理
- モデル更新（バージョンアップ、モデル変更）時の品質検証プロセス
- テスト資産の鮮度維持（定期的なレビューと更新）

## 章末の理解度チェック問題

1. マルチエージェントシステムのテストが従来のソフトウェアテストと異なる点を3つ挙げ、それぞれ説明せよ。
2. エージェント単体テストにおいて、LLMをモックに置き換える利点と限界を述べよ。
3. 統合テストでエージェント間のインターフェース整合性を検証する方法を説明せよ。
4. LLMの非決定性に対処するために、「抑制」と「許容」の2つのアプローチがある。それぞれ具体的な手法を2つずつ挙げよ。
5. 分散トレーシングの概念をマルチエージェントシステムに適用する場合、トレースID・スパンはそれぞれ何に対応するか説明せよ。
