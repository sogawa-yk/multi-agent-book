# 第5章 エージェント間の通信 ― MCP, A2A, そしてその先 ― 仕様書

## 章の概要

**章の目的**: マルチエージェントシステムにおけるエージェント間通信の課題と解決策を体系的に解説する。MCPの拡張的活用、A2Aプロトコルの詳細、メッセージ設計、同期/非同期通信の設計判断、エラー伝播の戦略を扱い、第4章の協調パターンを実現するための通信基盤を理解させる。
**目標ページ数**: 20p
**所属する部**: 第II部 マルチエージェントの理論

## 前提知識

- 第2章の内容（Function Calling、MCPの基本概念とアーキテクチャ）
- 第4章の内容（6つの協調パターン、特にオーケストレーター型とコレオグラフィ型）
- REST APIの基本概念
- 非同期処理の基本的な概念（コールバック、ポーリング等）

### 5.1 エージェント間通信の課題

**概要**: マルチエージェントシステムにおいて、エージェント同士がやり取りする際に発生する課題を整理する。エージェントの発見（どのエージェントが存在するか）、能力の記述（何ができるか）、メッセージの形式（どのように伝えるか）、通信の信頼性（確実に届くか）の4つの課題を提示する。

**図表**:
- なし（テキストで4つの課題を整理。後続の節で各課題への解決策を示す）

**キーポイント**:
- エージェントの発見: 分散環境でのサービスディスカバリに相当
- 能力の記述: エージェントが「何ができるか」を機械可読な形で公開する必要がある
- メッセージの形式: 非構造化テキストか構造化データか、スキーマの共有方法
- 通信の信頼性: ネットワーク障害、タイムアウト、リトライの設計

**概要**: 第2章で解説したMCPを、マルチエージェントの文脈で拡張的に活用する方法を解説する。MCPサーバーとしてエージェントを公開し、他のエージェントから「ツール」としてアクセスするパターンを示す。MCPの本来の用途（ツール提供）とエージェント間通信としての活用の違いを明確にする。

**キーポイント**:

- MCPの利点: 既存のエコシステム（MCPクライアント）をそのまま活用できる
- MCPの限界: 本来はツール提供のプロトコルであり、エージェント間の対等な通信は想定外
- 単方向のリクエスト/レスポンスが基本であり、双方向の協調には制約がある

### 5.3 A2Aプロトコル

**概要**: A2A（Agent-to-Agent Protocol）の詳細を解説する。Agent Card、Task、Message、Artifact、Streaming（SSE）の概念を体系的に説明し、A2AがMCPの限界をどのように補完するかを示す。A2AはGoogleが提唱したプロトコルであり、エージェント同士の発見・通信・協調を標準化する。

**図表**:
- 図5.1: MCP vs A2Aの比較図（役割・対象・通信モデルの違い）
- 図5.2: A2Aプロトコルのシーケンス図（Agent Card取得→Task送信→状態更新→Artifact返却）

**キーポイント**:
- Agent Card: エージェントの能力・エンドポイント・認証方式を記述するメタデータ
- Task: エージェント間のやり取りの単位（状態遷移: submitted → working → completed/failed）
- Message: Task内でやり取りされる個々のメッセージ（ロール: user/agent）
- Artifact: Taskの成果物（生成されたデータ、ファイル等）
- MCPとA2Aは補完的: MCPはツール提供、A2Aはエージェント間通信

- 出力形式の明示が結果の品質を左右する（構造化出力 vs 自由テキスト）
- コンテキストの受け渡し: 必要最小限の情報を渡す（コンテキストの過負荷を避ける）
- メッセージのバージョニングと後方互換性の考慮

**概要**: エージェント間の非同期通信パターンを解説する。長時間実行されるタスク（インフラ構築、大規模データ処理等）では同期的な通信が不適切であり、非同期通信（ポーリング、コールバック、イベント駆動）が必要となることを示す。

- 非同期通信における状態管理の必要性（タスクの進捗追跡）
- OCI固有のイベント駆動パターン（OCI Events、Notifications Service）

### 5.6 同期vs非同期の設計判断

**概要**: 同期通信と非同期通信の選択基準を整理し、ユースケースに応じた判断フレームワークを提示する。タスクの実行時間、結果の即時性要求、システムのスケーラビリティ要件等の観点から判断する。

**図表**:
- 表5.1: 通信パターン選択マトリクス（同期/非同期 x ユースケース x トレードオフ）

**キーポイント**:
- 同期通信: 短時間のタスク、結果の即時性が求められる場合
- 非同期通信: 長時間のタスク、スケーラビリティが求められる場合
- ハイブリッド: 初期応答は同期、進捗更新は非同期（A2AのStreamingパターン）
- 判断基準: タスクの予想実行時間、結果の即時性、システムの負荷特性

## 章間の接続

### 前の章（第4章: マルチエージェントの協調パターン）との関係
- 第4章で示した6つの協調パターンを実現するための通信メカニズムを本章で提供する
- オーケストレーター型やコレオグラフィ型は、本章で扱う通信設計が特に重要
- 第4章のパターンは「構造」、本章は「その構造を動かすための通信」

### 第2章（ツールとコンテキスト）との関係
- 第2章で解説したMCPの基本概念を前提とし、マルチエージェント文脈での拡張的活用を扱う
- MCPの3層アーキテクチャ（ホスト・クライアント・サーバー）の知識を活用する

### 次の章（第6章: 状態管理とメモリアーキテクチャ）への接続
- 本章で扱った通信の結果として生じる状態（タスクの進捗、中間結果）の管理を第6章で扱う
- 非同期通信における状態追跡の設計が、第6章の状態管理パターンに接続する
- 本章末で「エージェント間の通信メカニズムを理解した。次章では、通信の結果として生じる状態をどのように管理し、エージェント間で共有するかを掘り下げる」と接続する

### 第7章（設計原則）との関係
- エラーとタイムアウトの伝播（5.7節）は、第7章のフォールトトレランスの設計原則に直接接続する
