# 第6章 状態管理とメモリアーキテクチャ ― 仕様書

## 章の概要

**章の目的**: マルチエージェントシステムにおける状態管理の課題を体系的に整理し、共有メモリ・メッセージパッシング・永続化戦略の選択基準を読者が判断できるようにする。
**目標ページ数**: 18p
**所属する部**: 第II部 マルチエージェントの理論

## 前提知識

本章を読む前に、読者が理解している必要がある概念:

- **第2章**: コンテキストウィンドウの制約、短期記憶・長期記憶の基本概念
- **第4章**: 6つの協調パターン（特にオーケストレーター、コレオグラフィ）。パターンごとに状態管理の要件が異なることを理解していること
- **第5章**: エージェント間通信の基本。メッセージの形式と通信方式がわかっていること

## 節の構成

### 6.1 マルチエージェントにおける状態の種類

**概要**: マルチエージェントシステムで管理すべき「状態」を分類する。シングルエージェントでは会話履歴程度だった状態管理が、マルチエージェントでは多層化・複雑化することを示す。エージェントローカルな状態、タスク共有状態、グローバル状態の三層に分類し、それぞれの特性と管理方針を整理する。

**図表**:
- 図6.1: 状態の三層モデル（ローカル/タスク共有/グローバル）

**キーポイント**:
- エージェントローカル状態: 個々のエージェントが保持する会話履歴・中間推論結果
- タスク共有状態: 一つのタスクに関わる複数エージェントが共有する状態
- グローバル状態: システム全体で参照されるメタデータや設定情報
- 状態の粒度設計が、システム全体のスケーラビリティと一貫性を左右する

### 6.2 共有メモリ vs メッセージパッシング

**概要**: マルチエージェント間で状態を共有する二つの基本アプローチを比較する。ブラックボード型の共有メモリ方式と、メッセージパッシングによる状態伝搬方式の特性・トレードオフを整理する。協調パターンごとにどちらが適するかの指針を示す。

**図表**:
- 図6.2: 共有メモリとメッセージパッシングの比較図
- 表6.1: 協調パターンと状態共有方式の適合マトリクス

**キーポイント**:
- 共有メモリ（ブラックボード）: 全エージェントが同一データ構造にアクセス。実装がシンプルだが競合・整合性の問題が生じやすい
- メッセージパッシング: 状態をメッセージとして明示的に受け渡し。疎結合だが状態の全体像の把握が難しい
- オーケストレーター型は共有メモリと親和性が高く、コレオグラフィ型はメッセージパッシングと親和性が高い
- 実際のシステムではハイブリッド（両方の併用）が多い

### 6.3 短期共有コンテキスト

**概要**: タスク実行中に複数エージェント間で共有される短期的なコンテキストの設計を扱う。タスクコンテキスト（現在のタスクの進捗・中間成果物）と会話コンテキスト（対話の文脈）の管理方法を整理する。

**図表**:
- 図6.3: タスクコンテキストの構造と流れ

**キーポイント**:
- タスクコンテキスト: タスクの目標、進捗状態、中間成果物を構造化して管理
- 各エージェントに渡すコンテキストの「何を含め、何を除外するか」の設計が重要
- コンテキストウィンドウの制約を考慮した情報の要約・圧縮戦略
- 第2章で扱ったコンテキストエンジニアリングの原則がマルチエージェントでも適用される

### 6.4 長期記憶

**概要**: タスクの実行をまたいで永続化すべき知識・経験の管理を扱う。ベクトルDBによるセマンティック検索、構造化データとしての知識グラフ、経験ログの蓄積と活用方法を整理する。

**図表**:
- 表6.2: 長期記憶の実現方式比較（ベクトルDB / 構造化DB / ファイルストア）

**キーポイント**:
- ベクトルDBによるセマンティック検索: 類似した過去の経験・知識を検索
- 構造化データとしての知識管理: エンティティ間の関係を明示的に保持
- 経験記憶: 過去のタスク実行結果（成功・失敗）を蓄積し、将来の判断に活用
- マルチエージェントでは「どのエージェントが何を記憶するか」の設計が必要

### 6.5 状態の永続化戦略

**概要**: エージェントの状態をどのように永続化するかの戦略を整理する。スナップショット、イベントソーシング、チェックポイントの三つのアプローチを比較し、障害時の復旧や監査の観点から選択基準を示す。

**図表**:
- 図6.4: 状態管理アーキテクチャパターン図（スナップショット / イベントソーシング / チェックポイント）

**キーポイント**:
- スナップショット: 現在の状態全体を定期的に保存。シンプルだが状態変更の経緯が失われる
- イベントソーシング: 状態変更をイベントとして記録し、再生可能にする。監査・デバッグに強い
- チェックポイント: 処理の進捗ポイントを保存し、障害時にそこから再開する
- 長時間実行タスクでは、チェックポイントによる中断・再開が重要

### 6.6 コンテキストの分離と漏洩防止

**概要**: マルチエージェントシステムにおいて、あるエージェントの情報が意図せず別のエージェントに漏洩するリスクと、その防止策を扱う。テナント分離、権限管理、情報のスコープ制御の設計を整理する。

**図表**:
- 図6.5: コンテキスト分離の設計パターン

**キーポイント**:
- コンテキスト漏洩: あるエージェント向けの情報が、意図せず別のエージェントのコンテキストに混入するリスク
- 情報のスコープ制御: エージェントごとに参照可能な状態の範囲を明示的に定義
- テナント分離: マルチテナント環境では、異なるユーザー/組織の情報を厳密に分離
- 「最小権限の原則」をエージェントの状態アクセスにも適用する

### 6.7 OCI上の選択肢

**概要**: 本章で扱った状態管理の各パターンを、OCI上のサービスでどのように実現するかを整理する。Autonomous Database、Object Storage、Streaming、Cache with Redisなど、OCIが提供するサービスとの対応を示す。

**図表**:
- 表6.3: OCI上の状態管理オプション比較表（サービス / 用途 / 特性）

**キーポイント**:
- Autonomous Database: 構造化された状態の永続化、Vector Searchによる長期記憶
- Object Storage: 大容量の成果物・ログの保存
- OCI Streaming（Kafka互換）: イベントソーシングの基盤、エージェント間の非同期メッセージング
- OCI Cache with Redis: 短期共有コンテキストの高速アクセス
- 用途・パターンに応じた適切なサービスの選択指針

## 章末の理解度チェック問題

1. マルチエージェントシステムにおける状態の三層（ローカル/タスク共有/グローバル）を、それぞれ具体例を挙げて説明せよ。
2. 共有メモリ方式とメッセージパッシング方式のトレードオフを整理し、オーケストレーター型とコレオグラフィ型でそれぞれどちらが適するかを理由とともに述べよ。
3. イベントソーシングとスナップショットの違いを説明し、どのような場面でイベントソーシングが有利かを述べよ。
4. コンテキスト漏洩が発生するシナリオを一つ挙げ、その防止策を説明せよ。
5. OCI上で短期共有コンテキストと長期記憶をそれぞれ実現する場合、どのサービスを選択するか。理由とともに述べよ。
