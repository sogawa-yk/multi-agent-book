# 第9章 OCI上でシングルエージェントを構築する ― 仕様書

## 章の概要

**章の目的**: 第I部・第II部で学んだ理論とOCI GenAI Serviceの知識を活用し、OCI上で動作するシングルエージェントを設計・構築する方法を体系的に示す。第10章のマルチエージェント構築の基盤となる。
**目標ページ数**: 22p
**所属する部**: 第III部 OCI上での実践

## 前提知識

本章を読む前に、読者が理解している必要がある概念:

- **第1章**: エージェントの定義、ReActパターン
- **第2章**: Function Calling、MCP、コンテキスト管理
- **第7章**: 設計原則（単一責任、フォールトトレランス、冪等性、セキュリティ）
- **第8章**: OCI GenAI Serviceの利用方法、モデル選択、API呼び出しパターン

## 節の構成

### 9.1 アーキテクチャ設計

**概要**: OCI上でシングルエージェントを構築する際の全体アーキテクチャを設計する。エージェントの構成要素（LLM、ツール、メモリ、ランタイム）をOCI上のサービスにマッピングし、アーキテクチャの全体像を示す。

**図表**:
- 図9.1: シングルエージェントのアーキテクチャ図（OCIサービスへのマッピング）

**キーポイント**:
- 第1章で学んだエージェントの構成要素を、OCI上の具体的なサービスに対応づける
- LLM: OCI GenAI Service、ツール: OCI SDK / MCPサーバー、メモリ: Autonomous Database / Object Storage
- エージェントランタイムの配置先: OCI Container Instances / OKE / Compute
- アーキテクチャ決定記録（ADR）の考え方で設計判断を明示する

### 9.2 ReActループの実装

**概要**: エージェントの中核であるReActループ（推論→アクション→観察の繰り返し）の実装アプローチを示す。LLMへの問い合わせ、ツール呼び出しの判断、結果の統合というループの構造を整理する。

**図表**:
- 図9.2: ReActループのフロー図（推論→アクション選択→実行→観察→推論...）
- コード9.1: ReActループの基本構造（Python疑似コード）

**キーポイント**:
- ReActループの各ステップの役割と実装上の考慮点
- ループの終了条件: 目標達成、最大ステップ数、タイムアウト
- ツール呼び出し結果のコンテキストへの統合方法
- エラー発生時のループ内での対処（リトライ、スキップ、エスカレーション）
- コード例は構造理解のための疑似コードとして提示

### 9.3 ツールの実装

**概要**: エージェントが利用するツール（関数）の実装パターンを示す。OCI SDKを使ったOCIサービス操作ツール、外部API呼び出しツール、ファイル操作ツールなど、代表的なツールの設計と実装のポイントを整理する。

**図表**:
- 表9.1: ツールカテゴリと実装パターン
- コード9.2: OCI SDKを使ったツールの基本構造（Python疑似コード）

**キーポイント**:
- ツールの設計原則: 明確な入出力定義、冪等性、エラーハンドリング
- ツール定義（Tool Definition）の記述: 名前、説明、パラメータスキーマ
- LLMがツールを正しく選択するための説明文の書き方が重要
- OCI SDK操作をツールとしてラップする際のパターン
- コード例は構造理解のための疑似コードとして提示

### 9.4 MCPサーバーとしての公開

**概要**: 構築したシングルエージェントのツール群をMCPサーバーとして公開する方法を扱う。第2章で学んだMCPの概念を実装に落とし込み、他のエージェントやクライアントから利用可能にする。

**図表**:
- 図9.3: MCPサーバーのアーキテクチャと公開フロー

**キーポイント**:
- MCPサーバーの実装方式: stdio / SSE / Streamable HTTP
- ツールのMCPリソースとしての公開方法
- MCPサーバーのOCI上での配置（Container Instances等）
- 第10章でマルチエージェント化する際に、MCPサーバーが通信の基盤となることへの布石

### 9.5 認証の設計

**概要**: OCI上でエージェントがAPIを呼び出す際の認証設計を扱う。Resource Principal / Instance Principalによるキーレス認証、APIキーの管理、最小権限の原則に基づいたIAMポリシー設計を整理する。

**図表**:
- 図9.4: 認証フロー図（Resource Principal / Instance Principal / APIキーの使い分け）
- 表9.2: 認証方式の比較と推奨用途

**キーポイント**:
- Resource Principal: OCI上のリソース（Functions、Container Instances等）に割り当てる認証方式。APIキーの管理が不要
- Instance Principal: Computeインスタンスに割り当てる認証方式
- Dynamic Group + IAMポリシーによるアクセス制御
- 開発環境（APIキー）と本番環境（Resource Principal）の切り替え
- 第7章で学んだ最小権限の原則・セキュリティ設計の具体的な適用

### 9.6 エラーハンドリングとリトライ

**概要**: エージェント運用における各種エラーへの対処パターンを整理する。LLMのレスポンスエラー、OCI APIのスロットリング、ツール実行の失敗など、想定すべるエラーパターンとリトライ戦略を示す。

**図表**:
- 図9.5: エラーハンドリングのフロー図（エラー分類→対処判断→リトライ/フォールバック/停止）
- 表9.3: エラーパターンと対処戦略

**キーポイント**:
- エラーの分類: 一時的エラー（リトライ可能）vs 永続的エラー（対処変更が必要）
- LLM固有のエラー: レート制限、コンテキスト長超過、不正な出力形式
- OCI API固有のエラー: 429（Too Many Requests）、サービス制限、認証エラー
- 指数バックオフとジッターによるリトライ戦略
- 第7章で学んだフォールトトレランスの設計原則の具体的な適用

### 9.7 コンテキスト管理の実装

**概要**: エージェントのコンテキスト（会話履歴、ツール結果、システムプロンプト）の管理実装を扱う。コンテキストウィンドウの制約内で、必要な情報を効率的に管理する方法を整理する。

**図表**:
- 図9.6: コンテキスト管理の戦略図（優先度付きのコンテキスト構成）

**キーポイント**:
- コンテキストの構成要素: システムプロンプト、会話履歴、ツール結果、ワーキングメモリ
- コンテキストウィンドウの予算管理: 各要素にどれだけのトークンを割り当てるか
- 古い会話履歴の要約・圧縮戦略
- ツール結果が大きい場合のトリミング方法
- 第2章のコンテキストエンジニアリングの原則を、実装レベルに落とし込む

## 章末の理解度チェック問題

1. シングルエージェントの構成要素（LLM、ツール、メモリ、ランタイム）をOCI上のサービスにマッピングし、それぞれの選定理由を述べよ。
2. ReActループにおける終了条件として、どのような条件を設けるべきか。それぞれの条件が必要な理由とともに説明せよ。
3. Resource PrincipalとAPIキーの使い分けを、開発フェーズと本番フェーズの観点から説明せよ。
4. OCI GenAI Serviceの429エラー（レート制限）に対して、エージェントはどのように対処すべきか。リトライ戦略を含めて説明せよ。
5. コンテキストウィンドウの予算管理において、システムプロンプト・会話履歴・ツール結果にどのようにトークンを配分するか。優先順位の根拠とともに述べよ。
