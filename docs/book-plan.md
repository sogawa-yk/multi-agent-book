# 書籍企画書 (Book Plan)

## 書籍概要

### タイトル
**AIエージェント開発入門** ― シングルからマルチへ

### サブタイトル
OCI Generative AI Serviceで学ぶマルチエージェントの設計と実装

### 形態
- 判型: B5判
- 想定ページ数: 約300ページ
- 配布形態: 技術同人誌（技術書典向け）

### コンセプト
- **シングルからマルチへの段階的理解**: シングルエージェントの限界を構造的に理解した上で、マルチエージェントへの移行を自然に動機づける
- **理論と実践の両輪**: 協調パターン・通信プロトコル・状態管理の理論を学んだ後、OCI上での実装に落とし込む
- **図表ファースト**: 各節の核となるメッセージを図で表現し、本文は図の解説として構成する

### ビジョン
本書は、シングルエージェントの経験を持つOCI技術者が、マルチエージェントシステムを体系的に理解し、設計・構築できるようになるための実践ガイドである。「なぜマルチが必要か」の動機づけから始まり、協調パターン・通信プロトコル・状態管理の理論を経て、OCI Generative AI Serviceを中心とした実装まで、一貫した流れで学習できる。読了後には、ユースケースに応じたパターン選択からOCI上での実装、運用までを自力で行える知識と判断力を身につけることを目指す。

## 対象読者

### プライマリーペルソナ
- OCI技術者（クラウドエンジニア、インフラエンジニア、ソリューションアーキテクト）
- LLMを使ったアプリケーション開発の基礎知識がある（APIの呼び出し経験あり）
- エージェントの基本概念（LLM + ツール呼び出し）は理解済み
- マルチエージェントシステムを体系的に学びたい、実務で設計・構築する必要がある

### 前提知識
読者が本書を読む前に理解していることを前提とする:
- Pythonの基本的なプログラミングスキル
- OCIの基本的な操作（コンソール操作、CLI、SDK利用）
- LLMの基本的な概念（プロンプト、トークン、推論）
- REST APIの基本（HTTPメソッド、JSON）
- エージェントの基本概念（LLMにツールを持たせて自律的にタスクを実行する仕組み）

### 前提としない知識
本書で説明する:
- マルチエージェントの協調パターン（直列、並列、オーケストレーター型など）
- MCP（Model Context Protocol）の詳細な仕組み
- A2A（Agent-to-Agent）プロトコル
- OCI Generative AI Serviceの詳細な機能と使い方
- エージェント間の状態管理・メモリアーキテクチャ
- マルチエージェントシステムのテスト・運用手法

## 到達目標

読了後、読者が以下をできるようになること:

1. シングルエージェントの「どこに限界があるか」を構造的に説明できる
2. マルチエージェントの主要パターン（直列、並列、オーケストレーター、スーパーバイザー、コレオグラフィ、評価者ループ）を理解し、ユースケースに応じて選択できる
3. エージェント間の通信プロトコル（MCP, A2A）の役割と違いを整理できている
4. OCI Generative AI Service + OCIサービス群を使って、マルチエージェントシステムを設計・構築できる
5. マルチエージェントシステムのテスト・デバッグ・運用の勘所を押さえている

### 各章の概要

#### 第I部　シングルエージェントの深い理解

**第1章　AIエージェントとは何か ― 改めて定義する（14p）**
「エージェント」という曖昧な用語を正確に定義し、構成要素（LLM / プロンプト / ツール / メモリ / プランニング）を分解する。ReActパターン、エージェントランタイム、フレームワークの位置づけを整理する。

**第2章　ツールとコンテキスト ― エージェントの「手」と「記憶」（18p）**
Function Calling / MCP / コンテキスト管理を深く理解する。ツール定義のベストプラクティス、MCPのアーキテクチャ、コンテキストエンジニアリングまでをカバーする。

**第3章　シングルエージェントの限界 ― なぜマルチが必要になるのか（16p）**
単一エージェントの限界（コンテキスト爆発、専門性のジレンマ、信頼性の壁、並行処理の不在、責任分界の曖昧さ）を具体例で示し、マルチエージェントの動機を理解する。OKEクラスタ構築を1エージェントでやろうとする具体例を提示。

#### 第II部　マルチエージェントの理論

**第4章　マルチエージェントの協調パターン（24p）**
本書の理論的核心。直列パイプライン、並列ファンアウト/ファンイン、オーケストレーター型、スーパーバイザー型、コレオグラフィ型、評価者ループの6パターンを体系的に解説する。パターンの組み合わせとディシジョンツリーを含む。

**第5章　エージェント間の通信 ― MCP, A2A, そしてその先（20p）**
エージェント間通信のプロトコルと設計選択を理解する。MCPの拡張的活用、A2Aプロトコルの設計思想と仕組み、メッセージ設計、同期/非同期の判断を扱う。

**第6章　状態管理とメモリアーキテクチャ（18p）**
マルチエージェントにおける状態共有と永続化の設計を学ぶ。共有メモリ vs メッセージパッシング、短期/長期記憶、チェックポイント、コンテキストの分離を扱う。

**第7章　マルチエージェントの設計原則（16p）**
設計時の判断軸を持つ。単一責任、エージェント粒度設計、Human-in-the-Loop、フォールトトレランス、冪等性、セキュリティ、マルチエージェント採用の判断基準を学ぶ。

#### 第III部　OCI上での実践

**第8章　OCI Generative AI Service入門（20p）**
OCI上でLLMを利用するための基盤知識を固める。提供モデル（Cohere Command R+, Meta Llama等）、APIの基本、OCI Python SDKでの呼び出し、Function Calling対応状況を整理する。

**第9章　OCI上でシングルエージェントを構築する（22p）**
OCI GenAI Serviceを使ってエージェントを1つ作り、基盤を確立する。ReActループの実装、ツールの実装、MCPサーバー化、認証設計、エラーハンドリングを実装する。

**第10章　マルチエージェントシステムの構築（28p）**
本書の実践的核心。複数エージェントを協調させるシステムをOCI上に構築する。エージェント分割設計、オーケストレーター実装、通信実装、共有状態実装、Human-in-the-Loop実装を行う。

**第11章　OCI固有のサービスを活用するエージェント設計（20p）**
Resource Manager、OKE、Autonomous Database、Streaming/Queue、Logging/Monitoring、Vault、API Gatewayを、エージェントのツール/基盤として活かす設計を学ぶ。

**第12章　ケーススタディ ― OKEクラスタの自動構築システム（24p）**
第I〜III部の知識を統合する。ヒアリング→アーキテクチャ設計→IaC生成→デプロイ→検証の一連の流れを、マルチエージェントシステムとして設計・実装する。

#### 第IV部　運用と発展

**第13章　マルチエージェントのテストとデバッグ（18p）**
マルチエージェント特有のテスト・デバッグ手法を学ぶ。単体テスト、統合テスト、E2Eテスト、LLMの非決定性への対処、トレーシングを扱う。

**第14章　可観測性とガバナンス（16p）**
本番運用に耐える監視・ガバナンスの設計を学ぶ。ログ設計、メトリクス、アラート、コスト管理、ガバナンスを扱う。

**第15章　マルチエージェントの未来とエコシステム（12p）**
現在の限界と今後の方向性を展望する。標準化の動向（MCP / A2A）、フレームワークの進化、OCIのロードマップを整理する。

## LLM環境

- **使用サービス**: OCI Generative AI Service
- **利用可能モデル**: Cohere Command R+、Meta Llama等（OCI上で提供されるモデル群）
- **コード例の言語**: Python（OCI SDK for Python）。ただしコード例は概念理解の補助であり、実装の主体はClaude Code等のAIツールを想定

## 執筆方針

- **図表ファースト**: 各節の核となるメッセージは図で表現し、本文は図の解説として書く。各節に最低1つの図表を配置する。図解にはMermaid記法を使用する
- **OCI環境前提**: コード例・構成例はすべてOCI Generative AI Service + OCI SDKで統一する
- **コードは理解の補助**: コード例は設計概念の理解を助けるために必要な場合のみ掲載する。読者は本書で設計・理論を学んだ後、Claude Code等のAIツールを使って実装する想定のため、写経用の完全実装は載せない
- **既存フレームワークへの言及**: LangChain, CrewAI, AutoGen等は概念の理解を助けるために参照するが、特定フレームワーク依存にはしない
- **各章末に理解度チェック**: 3〜5問の確認問題で、読んだ内容の定着を促す
- **実体験に基づくリアルな設計判断**: Galley開発の知見を直接的に言及せず、ケーススタディに反映する

## スコープ外

明示的にスコープ外とする項目:
- 特定のエージェントフレームワーク（LangChain, CrewAI等）の詳細なチュートリアル
- OCI以外のクラウドプラットフォーム（AWS, Azure, GCP）での実装
- LLMのファインチューニング・トレーニング
- プロンプトエンジニアリングの網羅的な解説
- フロントエンドUIの実装
- 商用環境向けの本番デプロイ手順（本書はプロトタイプ・検証環境レベル）

## 成功指標

- 読者がマルチエージェントの6つの協調パターンを説明でき、ユースケースに応じた選択ができる

- 読者がMCPとA2Aの役割と違いを第三者に説明できる
- 第3章までの読了で「なぜマルチが必要か」を自分の言葉で説明できる
- 第12章のケーススタディを通じて、現実的な設計判断のトレードオフを理解できる
